## Full suite extension (added same session)

After the initial five-pass sweep I went back and ran the tests the first pass skipped. All green, with one new bug:

**Pass 1 extension — idempotency.** `dpkg --purge → apt install → dpkg --purge → apt install` cycle: clean every time. `/var/lib/aaos` and user + group removed on purge, re-created on install.

**Pass 2 extension — operator access.**
- Non-root user in `aaos` group → `agentd list` succeeds (returns `No agents running.`).
- Non-root user NOT in `aaos` group → `agentd list` fails with an excellent error: `"Are you in the aaos group? ... sudo adduser $USER aaos (log out + in)"`. Actionable.
- `systemctl stop agentd` → socket removed cleanly.
- `systemctl start agentd` → socket restored, owner + mode intact.
- Malformed `/etc/default/aaos` (added a `=bad syntax` line): daemon still starts — systemd `EnvironmentFile=-` with leading `-` tolerates malformed lines silently. Arguable UX gap (no warning emitted), but not a regression from v0.0.1.
- `dpkg --purge` while daemon running: systemd stops the service cleanly via `prerm`, socket removed, no orphan processes in `ps aux`.

**Pass 4 extension — destructive fault injection.**
- `kill -9 <MainPID>`: systemd logs `Main process exited, code=killed, status=9/KILL` → `Failed with result 'signal'` → 5s `RestartSec` delay → `Started agentd.service` → submit works again. Operator sees the chain in `journalctl`.
- OOM via `MemoryMax=20M` override: 20 MB turned out not tight enough to hit the ceiling on a simple plan; plan ran degraded (multiple web_fetch retries) and returned via timeout. Daemon stayed `active`. Stronger cap would kill cleanly but this soft-degraded path is also fine.
- Actual disk-full (`tmpfs` of 100 KB, 90% pre-filled): `file_write` returns `No space left on device (os error 28)`, agent retries once, second write fails identically, plan terminates cleanly. Daemon survives.

**Pass 5 extension — skill shadowing + external MCP client.**
- Operator drops `/etc/aaos/skills/context-engineering/SKILL.md` shadowing the bundled `/usr/share/aaos/skills/context-engineering/SKILL.md`. Verified via code inspection: `SkillRegistry::new` uses `HashMap::insert`, which last-wins. `discover_all_skills` iterates `/usr/share/` → `/etc/aaos/` → `/var/lib/` → `AAOS_SKILLS_DIR` in that order, so operator skills override bundled ones correctly.
- External MCP client registration: spawned a mock stdio MCP server (trivial bash script responding to `initialize` + `tools/list`), configured it in `/etc/aaos/mcp-servers.yaml`. Startup log: `MCP client: 1 of 1 server(s) registered`. Handshake + tool-list handshake both succeeded; mock server's `echo` tool is now available to aaOS agents as `mcp.mock.echo`.

## One new bug surfaced during the extended pass

**Bug 7 (new)** — `memory_query` tool routes `[worker]` but isn't in the worker-side whitelist (`aaos_backend_linux::worker_tools::WORKER_SIDE_TOOLS`). Calling it produces `tool error: tool memory_query not available in worker not available in worker` (the "not available in worker" substring is also duplicated — stringly-constructed error message bug).

Also `memory_store` + `memory_delete` almost certainly have the same issue by symmetry (they were added to the tool registry but not to the worker whitelist when the worker-side routing was introduced).

**Severity:** Medium. Agents with memory-using capabilities fail cleanly (with a legible error) instead of silently succeeding, which is the right failure mode. But it means memory-tool agents don't work under confinement at all today. Fix should be routine: add `memory_store`/`memory_query`/`memory_delete` to `DAEMON_SIDE_TOOLS` in `aaos-core::tool_surface` (they're daemon-owned anyway — backend needs LLM embedding access). Not a v0.0.2 blocker (v0.0.2 never advertised confined memory tools), but ship-worthy in v0.0.3.
