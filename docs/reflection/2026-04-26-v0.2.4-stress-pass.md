# 2026-04-26 — Stress probe on v0.2.4 (Bug 35 — first concurrency finding)

Round 9 + the fuzz pass both depleted v0.2.x's source-reading + input-handling surfaces.  The third probe — multi-threaded stress — was named in v0.2.4's CHANGELOG as the next probe of different kind.  First serious run found Bug 35.

## Setup

- Same droplet as rounds 5–9 (a fresh DigitalOcean Debian 13 droplet).
- v0.2.4 `.deb`, InProcessBackend.
- `/root/stress-droplet.sh` — bash + socat + jq harness that opens N concurrent connections to the daemon's Unix socket and hammers `agent.spawn` + `agent.stop` cycles.  Each cycle spawns an on-demand agent with a minimal manifest (`tool: echo` capability only — no LLM round-trip), then stops it.  Watchdogs: journald scan for panic/BUG/deadlock, agent.list for orphans, `aaos-agent-worker` zombie count, daemon RSS delta.
- 32 workers × 500 spawns each = 16,000 spawn-stop cycles per pass.

## What it found

Three consecutive 16k-cycle passes on the same daemon:

| Pass | RSS before | RSS after | Δ |
|---|---|---|---|
| 1 | 19,700 KB | 40,316 KB | **+20,616 KB** |
| 2 | 40,316 KB | 58,424 KB | **+18,108 KB** |
| 3 | 58,424 KB | 78,260 KB | **+19,836 KB** |

Linear ~19MB/16k-cycles ≈ 1.2 KB per spawn-stop cycle.  Sustained.  After 48k cycles, RSS quadrupled (20→78MB).  No alive agents in the registry between passes — the leak isn't the agent table.

## Root cause

`InMemoryAuditLog::new()` is unbounded by design — its own doc-comment says *"Unbounded by default; opt-in cap via `with_cap()` for long-running test harnesses where unbounded growth would OOM."*  Both `Server::new()` and `Server::with_llm_and_audit()` constructed it via `::new()`.  Each spawn-stop cycle emits ~10 audit events (`AgentSpawned`, `CapabilityIssued` × N, `AgentStopped`, `CapabilityRevoked` × N, `BudgetTrackerCreated`, etc.) at ~120 bytes each.  10 × 120 = 1.2 KB/cycle — matches the observed leak rate exactly.

Why the source-reading loop missed this for nine rounds: the bug isn't in source the agent reads as suspicious.  `Server::new()` calls `::new()` on the audit log, and the doc-comment that says "unbounded" reads like documentation, not a warning.  The bug is *only visible under sustained load* — exactly the surface the stress probe covers and the other probes can't.

Why the fuzz pass missed it: fuzzers find panics on arbitrary inputs.  This is a slow leak, not a panic on input — fuzzer would need a multi-call workload generator + RSS measurement, which is essentially what the stress harness is.

## Fix

Both Server constructors now call `InMemoryAuditLog::with_cap(Self::audit_log_cap())`.  Default cap 50_000 events (~6MB), override via `AAOS_AUDIT_LOG_CAP`.  At 50k events ÷ 10 events/cycle ≈ 5_000 spawn-stop cycles of recent history.

`with_cap` already existed for tests (Bug 33's always-on assert lives there); production just wasn't using it.

## Verification

Same 3-pass stress on v0.2.5 droplet:

| Pass | RSS before | RSS after | Δ |
|---|---|---|---|
| 1 | 19,912 KB | 35,512 KB | +15,600 KB *(initial fill of the 50k cap)* |
| 2 | 35,512 KB | 35,620 KB | **+108 KB** |
| 3 | 35,620 KB | 35,632 KB | **+12 KB** |

Pass 1 grows because the VecDeque fills to capacity.  Passes 2 and 3 are essentially flat — `pop_front()` is reusing slots.  Memory bounded; leak closed.

## Probe coverage now

Three independent probes, three different bug classes:

| Probe | Round | Bugs found | Surface covered |
|---|---|---|---|
| Source-reading reflection | 6–8 | 27–34 (8 real) | Logic / spec drift / capability bypass |
| Fuzzing (libfuzzer) | 9-companion | 0 / 137M inputs | Panic on input, deserialize edge cases |
| Concurrency stress | 9-companion | **35 (1 real)** | Long-lived state / leaks under churn |

Different probes, different findings.  The pattern that started in round 6 (deferred-follow-up code comments are findings) holds for stress too: `InMemoryAuditLog`'s "Unbounded by default" doc-comment is the same shape — a known property *documented* but not handled in production callers.

## Cost

- Setup: ~30 min (write harness, debug heredoc/jq quoting, fix `lifecycle: bootstrap` → `on-demand`, fix `pgrep -fc` newline, set up before/after RSS capture).
- Run: 3 passes × 100s = 5 min (initial leak detection); 3 more passes after fix to verify = 5 min.
- Total: ~40 min from "design probe" to "Bug 35 fixed and verified."

## Caveats and what stress did NOT find

- The probe tested **InProcessBackend** only.  A second pass under `AAOS_DEFAULT_BACKEND=namespaced` would exercise the broker `SessionMap` + worker process churn — different concurrency surfaces (Unix-socket accept storm, fork/exec under namespace setup, broker reader/writer task lifetimes).  Queueing as a follow-up.
- Single-host stress.  Multi-tenant / multi-daemon stress is out of scope (no multi-host runtime today).
- Manifest used `lifecycle: on-demand` with no LLM tools.  A stress run that submits real LLM-using goals would exercise the executor + scheduler concurrency surfaces (different from the registry-level surfaces this run hit).  Cost: ~$0.01 per spawn × 16k = $160; not worth it for a routine stress without a specific signal driving it.

## Follow-ups

1. **Stress under namespaced backend** — same harness, `AAOS_DEFAULT_BACKEND=namespaced` set on the droplet.  Different surfaces (worker fork/exec under landlock + seccomp setup, broker `SessionMap`).  Probably finds nothing (the namespaced path has been exercised by every soak test since v0.0.2 droplet QA), but worth confirming.
2. **Wire stress into CI** — the harness is bash + socat, runs in <2 min on a small instance.  Catches regressions where a future commit re-introduces a leak.  Filing as a CI-improvement task.
3. **Audit-log-cap tunable surfaced in operator docs** — `AAOS_AUDIT_LOG_CAP` deserves a mention in `man agentd` and the operator guide; deferred-by-design until M1's image-build needs it.

## Pattern lifted

> **A code comment that calls itself a "default" is a finding waiting to happen if production callers use it.**
>
> `InMemoryAuditLog::new()`'s "Unbounded by default" was honest documentation in the test-harness sense (memory-only test, why bound it?) but became a production bug because both `Server::new()` and `Server::with_llm_and_audit()` called `::new()` directly.  The convention should be: if a primitive's default is *unsafe for production* (unbounded, blocking-on-default-timeout, no-rate-limit), the doc-comment should *direct production callers to the bounded variant* — and ideally, the unbounded `::new()` should be `#[cfg(test)]`-gated.  Tightening `InMemoryAuditLog::new()` itself is queued as v0.2.6+ work.
