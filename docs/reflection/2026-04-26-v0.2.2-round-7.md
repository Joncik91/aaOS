## What the run exposed

### Bug 31 — `BudgetTracker` reset race (medium)

`crates/aaos-core/src/budget.rs::maybe_reset` did `let last = ...load(...); ...last_reset_check.store(now, ...)` with no atomicity between the two.  Two threads near a period boundary could both pass the `now - last >= 1` gate, both store, and both call `reset()` — clobbering tokens already tracked in the new period.  Fixed by replacing load+store with a CAS loop on `last_reset_check`.  Test `maybe_reset_races_have_at_most_one_winner` hammers `track()` with 16 threads × 100 calls and asserts cumulative `used` matches cumulative successful tracks.  Commit `8c06449`.

### Bug 32 — `safe_open_for_capability` intermediate-component symlink TOCTOU (high)

`crates/aaos-tools/src/path_safe.rs::safe_open_for_capability` used `open()` with `O_NOFOLLOW` — but `O_NOFOLLOW` only protects the *leaf* component.  An attacker who could swap any intermediate directory for a symlink would steer the open elsewhere.  v0.2.2's `file_write.rs` comment explicitly disclaimed this as "out of scope of this fix."

Fix: route through `openat2(AT_FDCWD, path, OpenHow{flags, resolve: RESOLVE_NO_SYMLINKS})` which rejects symlinks at *every* path component.  Available since Linux 5.6 (Debian 13's 6.12+ kernel has it).  Falls back to plain `open()` with `O_NOFOLLOW` on older kernels (`ENOSYS`) or on hosts where seccomp denies the syscall (`EPERM`) — leaf-only protection same as v0.2.2.

Two `openat2` API quirks worth recording for future maintenance:
- `O_NOFOLLOW` must be stripped from the flags when calling `openat2(RESOLVE_NO_SYMLINKS)`; the kernel returns `EINVAL` if both are set together (the resolve flag strictly subsumes O_NOFOLLOW).
- `open_how::mode` must be 0 unless `O_CREAT` or `O_TMPFILE` is set.  Passing a non-zero mode for a read-only open returns `EINVAL`.

Worker seccomp policy gains `SYS_openat2`.  Test `intermediate_component_symlink_rejected` plants a symlink as a parent dir and asserts the open is refused.  Commit `67e7d24`.

### Bug 33 — `InMemoryAuditLog::with_cap(0)` deadlock (medium)

`crates/aaos-core/src/audit.rs:309` used `debug_assert!(max >= 1)` — compiled out in release.  With `max == 0`, `record()`'s `while events.len() >= max` loop is always true and `pop_front()` on an empty `VecDeque` is a silent no-op.  Result: infinite spin while holding the audit mutex, deadlocking every recorder thread.

Fix: switch to always-on `assert!`.  Misuse fails loud at construction.  Test `in_memory_audit_log_with_cap_zero_panics`.  Commit `627846e`.
