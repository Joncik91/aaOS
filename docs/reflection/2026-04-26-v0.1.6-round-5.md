## What the run exposed — Bug 27 (real, shipped)

`crates/agentd/src/spawn_tool.rs` — both first-attempt (line 215-219) and retry (line 281-286) paths called `CapabilityToken::issue(child_id, child_cap, Constraints::default())` to issue child tokens.  The parent's `max_invocations`, `rate_limit`, and `expires_at` were silently dropped.  A parent with `WebSearch { max_invocations: Some(1) }` could spawn a child with `web_search` capability; the child's token had no invocation cap.  Phase A's original fix had regressed in a later refactor.

The reason the spawn paths bypassed `narrow()`: the existing `CapabilityToken::narrow()` only tightens constraints — it doesn't allow capability substitution.  So when the parent holds `file_read: /src/*` and the child asks for `file_read: /src/crates/*`, `narrow()` would preserve the parent's broader scope.  The workaround issued fresh tokens with `Constraints::default()`, bypassing the issue but losing constraint inheritance.

**Fix shape**: added `CapabilityToken::narrow_with_capability(child_agent, child_capability, additional)` that (a) verifies the child's requested capability is a subset of the parent's via `capability_matches()`, (b) clones the parent token preserving its constraints, (c) substitutes the narrower capability and resets issuance metadata, and (d) layers any additional constraints on top.  Plus a registry wrapper.  Both spawn paths refactored to use the new method.
