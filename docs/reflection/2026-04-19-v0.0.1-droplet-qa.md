# 2026-04-19 — v0.0.1 droplet QA

Soak-test of the v0.0.1 `.deb` on a fresh DigitalOcean Debian 13 droplet (4 vCPU / 8 GB, kernel 6.12.43, Landlock active in the LSM stack). Goal: surface packaging, daemon-lifecycle, and agentic-surface bugs before M1 (image) starts.

## Setup

- Fresh droplet, clean `apt-get update` baseline.
- Installed `lintian` for static package analysis.
- Two .deb builds tested: (a) the first build shipped only `--features mcp`; (b) a rebuild added `--features mcp,namespaced-agents` after bug #1 was found.
- DeepSeek key stored at `/etc/default/aaos` mode 0600 root:root; keyed via direct file edit (QA did not exercise `agentd configure`'s interactive path because stdin-on-ssh is awkward; the `--key-from-env` non-interactive path is unit-tested locally).
- Canonical goal: *"fetch https://news.ycombinator.com and list the top 5 story titles, write to /data/hn.md"*.

## What passed

- **Package hygiene structurally sound.** `dpkg-deb -c` lists expected assets (75 files), skills directory contains 21 skills, `mcp-servers.yaml.example` present, roles all conffile-marked. Purge cleanly removes `/var/lib/aaos/`, the `aaos` user, and the `aaos` group.
- **Install → start path works.** `apt install` brings up `agentd.service` within 2 s, systemd reports `active (running)`, socket at `/run/agentd/agentd.sock` mode 0660 owned `aaos:aaos`.
- **Postinst kernel probe fires correctly.** On this kernel (Landlock + userns both available) the generated `/etc/default/aaos.example` uncomments `AAOS_DEFAULT_BACKEND=namespaced` and `AAOS_CONFINE_SUBTASKS=1` with the expected "Kernel probes passed" comment.
- **Fresh missing-key warning fires.** Startup without a key prints *"No LLM client configured — neither DEEPSEEK_API_KEY nor ANTHROPIC_API_KEY is set. Run `sudo agentd configure` to set one…"* — the M1 change landed correctly.
- **Confinement works end-to-end (once the build had `namespaced-agents`).** Canonical goal ran in **41 s**, writer agent's `file_read` + `file_write` calls tagged `[worker]` in the audit stream, fetcher (scaffold, by design) stayed `[daemon]`. `/data/hn.md` contained a real current story ("Vercel April 2026 security incident", 280 points) — not training-data fabrication.
- **Corrupt role YAML — excellent error path.** Replacing `writer.yaml` with garbage produced `ERROR: role YAML failed to parse — this role will not load. Fix the file or remove it, then restart agentd. file=… error=found character that cannot start any token at line 1 column 20`. Exact file, line, column, next-step. Keep.
- **Invalid key fails fast.** Bad DeepSeek key causes the submit to fail within 1 s instead of hanging. (But silently — see Bug 3.)
- **MCP server loopback works.** `tools/list` against `127.0.0.1:3781` returns the three expected tools (`submit_goal`, `get_agent_status`, `cancel_agent`) with correct JSON schemas.
- **Workspace GC sweeps on startup.** Log line shows `removed=N total=M` every boot.

## What broke

### Bug 1 — CRITICAL — `.deb` shipped without `namespaced-agents` feature

The 2026-04-19 build-deb.sh change enabled `--features mcp` but not `namespaced-agents`. The shipped v0.0.1 .deb had `NamespacedBackend` stripped out at compile time, so `AAOS_DEFAULT_BACKEND=namespaced` in `/etc/default/aaos` (both manual and postinst-generated) silently fell through to `InProcessBackend`. Every tool call audit tag read `[daemon]` regardless of env.

**Evidence.** `strings /usr/bin/agentd | grep NamespacedBackend` returned empty on the first build; after rebuild with `--features mcp,namespaced-agents` the binary gained `using NamespacedBackend` + `aaos-agent-worker` strings and the audit tags flipped to `[worker]` where expected.

**Fix.** `packaging/build-deb.sh` default `AAOS_BUILD_FEATURES="mcp,namespaced-agents"`. Landed as part of the QA session.

**Why this matters.** This was the whole M1 promise — "confinement by default on capable kernels." The feature probe in postinst told operators they had it; the binary didn't deliver. A release announcing "Landlock + seccomp subtask confinement by default" while shipping a binary that can't do it is worse than not promising it at all.

### Bug 2 — MEDIUM — Zombie `aaos-agent-worker` processes

After a successful run, `ps auxf` shows `[aaos-agent-work] <defunct>` children of `agentd`. The worker process exits cleanly but the parent doesn't reap it. Accumulates over time and will eventually hit PID-table ceilings on long-lived daemons.

**Evidence.** One zombie after the first namespaced run; two after the second; probably one per successful worker launch.

**Likely root cause.** Somewhere in `crates/aaos-backend-linux/src/lib.rs` around the launch/stop path the `waitpid` after broker close doesn't run on the happy-exit path — the 2026-04-19 `waitpid` watchdog was added for fail-fast on *early* exit, not for post-success reaping.

### Bug 3 — MEDIUM — Invalid API key fails silently from operator's POV

A wrong/expired LLM key produces `bootstrap failed (0k in / 0k out, 0s)` with **zero log lines** explaining why. `agentd submit -v` adds nothing. Journalctl at the same timestamp contains no error.

**Expected.** An operator-visible ERROR log naming the LLM provider's HTTP status (401, 403) and a hint pointing at `/etc/default/aaos`.

**Why this matters.** #1 first-run failure mode for new operators is a mistyped key. The current UX forces them to guess.

### Bug 4 — MEDIUM — MCP client failures completely silent

With a `client.servers` entry pointing at an unreachable URL (`http://127.0.0.1:9999`), daemon startup + restart produces **zero** log lines mentioning MCP at all. No "connecting to…", no "failed to reach…", no "reconnecting in Xs…". The mcp-servers.yaml parse presumably succeeds and the reconnect loop swallows errors.

**Expected.** INFO line for each configured MCP server at startup ("attempting to connect to mcp://…"), WARN on first failure, DEBUG on subsequent reconnect attempts.

### Bug 5 — LOW — Duplicate backend-selection log line

The "AAOS_DEFAULT_BACKEND=namespaced — using NamespacedBackend" INFO line prints **twice** every startup. Probably one call from `Server::new` / `Server::with_llm_client` path and once from `build_in_process_backend`/`build_namespaced_backend` — two paths log the same fact.

### Bug 6 — LOW — Lintian surfaces 11 errors

`lintian` against the final .deb raises 11 errors. Triage:

| Class | Count | Severity | Fix cost |
|---|---|---|---|
| `aliased-location` (`lib/systemd/` → should be `usr/lib/systemd/`) | 4 | Real — Debian Trixie enforces merged-usr | Low — one path change in `[package.metadata.deb].assets` |
| `copyright-file-contains-full-apache-2-license` + `copyright-not-using-common-license-for-apache2` | 2 | Real — Debian Policy 12.5 | Low — trim `LICENSE` in `license-file` to a short reference |
| `description-starts-with-package-name` | 1 | Real — Debian Policy 3.4.1 | Low — tweak `description` in `[package.metadata.deb]` |
| `embedded-library libyaml` | 1 | Expected — Rust static linking | Override with a lintian rule; not a bug |
| `no-changelog` | 1 | Real — non-native packages need `changelog.Debian.gz` | Medium — add a tracked changelog generator |
| `unstripped-binary-or-object` | 2 | Real — release binaries should strip debug info | Trivial — `cargo-deb` has a flag; we're not using it |

Plus 9 warnings (recursive chown, `systemctl` calls in maintainer scripts without the expected debhelper wrappers, manpage-missing for `aaos-agent-worker`, gzip timestamp non-deterministic). All addressable; none block ship.

## What didn't get exercised

Pass 4 (fault injection) was partial — I ran the invalid-key + corrupt-YAML + unreachable-MCP cases but didn't hit:

- `kill -9` the daemon mid-run (one attempt had a PID race and didn't actually kill anything; re-try needed).
- OOM-killer scenario (needs cgroup memory limit setup).
- Disk-full on `/var/lib/aaos/`.
- LLM provider 429 rate-limit path.
- Child agent `kill -STOP` → TTL cascade verification.

Pass 5 (MCP client + skills exercise) only did the server-side `tools/list` smoke. Client-side (registering an external MCP server, calling through it) wasn't tested, and neither was the shadowing-skill behavior from the `discover_all_skills` change.

Plan: file follow-up tickets for Bugs 1–6, fix Bug 1 first (critical), then re-do Pass 4 fully after fixes land, then ship v0.0.2 with the image (M1) blocked on v0.0.2.

## Time and cost

- ~45 minutes wall clock.
- ~$0.02 DeepSeek (one 57s run + one 41s run + one 1s failure).
- Two full release-mode `cargo build` cycles (one per .deb rebuild), ~5 minutes each on A8.

## Verdict

**v0.0.1 as currently tagged on GitHub is misleading** — the attached .deb doesn't deliver confinement. A v0.0.2 bump that lands the `namespaced-agents` feature fix + addresses Bugs 3, 4, 5 is the honest next step before M1 starts on the image pipeline. Bug 2 (zombie reap) and Bug 6 (lintian hygiene) can slip to v0.0.3 or the image pass; neither blocks "does the .deb do what the README says."
