## What the run exposed

### Bug 38 — `SessionStore::clear()` has no production caller

`SessionStore::clear` exists on the trait but is invoked only from tests.  `persistent_agent_loop`'s exit handler doesn't call it.  `AgentRegistry::stop_sync` doesn't call it.  The `replace()` method (added v0.2.2 for atomic session rewrites) doesn't internally call `clear()` because `JsonlSessionStore::replace` is overridden to do an atomic-rename without the truncate step.  Result: every persistent agent that processes a message leaves its `(agent_id, Vec<Message>)` entry behind in the in-memory store on stop, accumulating unbounded across spawn-stop cycles.

The agent's report flagged the JSONL store specifically (zero-byte files leak), but the production daemon uses `InMemorySessionStore` — which has the same structural issue but stores in-memory rather than on-disk.  Both leak; only the in-memory path is reachable today.

**Fix shape**: gate cleanup on `persistent_identity`.  Bootstrap-shaped agents (stable id from `/var/lib/aaos/bootstrap_id`) need history to survive across stop+respawn so memory persists.  Ephemeral agents (no stable id) lose access to their `agent_id` after stop, so the history is unreachable — pure leak.  Added `clear_session_on_exit: bool` parameter to `persistent_agent_loop`; production callers (`AgentRegistry::start_persistent_loop`, `InProcessBackend::launch`) set it to `!persistent_identity`.  Tests pass `false` to preserve their assertion that history outlives loop exit.  Also fixed `JsonlSessionStore::clear` to `fs::remove_file` rather than truncate (defensive, even though the JSONL store has no production caller today).

### Bug 40 — `agent.spawn_and_run` + persistent leaks the persistent loop

`handle_agent_spawn_and_run` sequence:
1. `handle_agent_spawn(params)` — for `lifecycle: persistent`, this calls `backend.launch(spec)` which starts an in-daemon persistent task (and, under namespaced, a worker subprocess).
2. `execute_agent(agent_id, manifest, message, id)` — runs a one-shot `AgentExecutor::run` against the same agent_id.

Two concurrent execution loops on one agent.  After the one-shot returns, the persistent loop is never told to stop — it sits in `tokio::select!` waiting for `message_rx.recv()` or `command_rx.recv()`, neither of which ever arrives for ephemeral one-shot callers.  Per call: one tokio task leak (InProcess) or one `aaos-agent-worker` subprocess leak (namespaced — Bug 37 closed the leak for `agent.stop` but not for spawn_and_run that never calls stop).

**Fix shape**: spawn_and_run is a one-shot API; persistent agents need explicit lifecycle management via `agent.spawn` + `agent.run`.  Reject persistent manifests at the top of `handle_agent_spawn_and_run` with a clear error directing the caller.  Cheaper than building a "spawn one-shot, then auto-stop" code path that would replicate registry+backend cleanup logic.

### Bug 39 — `JsonlSessionStore::load_archives` / `prune_archives` scan whole directory (deferred)

Real structural inefficiency: both methods `read_dir` `self.data_dir` on every call and filter by filename prefix.  O(N) where N = total agents ever, not just the target agent's archives.  Production exploit doesn't fire because production uses `InMemorySessionStore`; `JsonlSessionStore` has no production caller today.

Filed in `docs/ideas.md` with reconsider signals: (a) durability requirement promotes JSONL to production, OR (b) multi-restart history retention becomes a feature.  Until either fires, refactoring an unused code path is premature.

## Process notes

- **Round 9's "depletion" was prompt-shape depletion, not surface depletion.**  Round 9 + the fuzz pass + the v0.2.5 InProcess stress all said "v0.2.x patch surface is depleted."  The namespaced stress probe (Bugs 36, 37) plus round 10 (Bugs 38, 40) closed FIVE more bugs in v0.2.5–v0.2.7.  Two of those (36, 37) needed a different probe type; the other three (35, 38, 40) needed the same probe type with a *better prompt*.  The lesson: "no findings" from a self-reflection pass is a depletion signal *for the specific prompt shape used*, not for the runtime.  Future rounds should explicitly try multiple prompt shapes before declaring depletion.
- **Test code that asserts behavior the production code doesn't actually achieve is a finding shape worth grepping for.**  `persistent_loop_processes_message` asserted that session history outlives loop exit.  But production callers had no path that depended on that — Bootstrap reloads via `session_store.load(agent_id)` at startup, not at stop.  The test was checking an invariant the production code was already violating (in spirit, since clear was never called) and that no production caller relied on (since ephemeral agents lose their id on stop anyway).  This is the same shape as "trait method with only test callers" — testing a contract no production caller actually invokes.

Earlier stress probes used messageless agents so this leak wasn't observable; a "spawn + send 1 message + stop" probe under v0.2.7 would confirm Bug 38's fix and incidentally test whether any other per-message state leaks.
