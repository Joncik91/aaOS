# Phase F-a — `agentd` as a Debian package *(2026-04-15)*

**Integration commits:**
- `5717906` "package: add Phase F-a .deb packaging scaffold"
- `8d45691` "capability-registry: add token_id_of for release builds"

**Setup.** Fresh Debian 13 cloud VM (kernel 6.12.43). Built Rust toolchain + `cargo-deb@2.7.0` on the VM, cloned the repo, built release artifacts, ran `cargo deb -p agentd`. Verified the `.deb` by `apt install` → `systemctl start agentd` → socket live → JSON-RPC responds → `apt purge` → user + state gone.

**What worked.** The `cargo-deb` approach ate the entire `debian/` scaffold — metadata lives in `[package.metadata.deb]` on the `agentd` crate, maintainer scripts in `packaging/debian/`, unit in `packaging/`. No hand-maintained control/rules/changelog tree. Output is a single 3.4MB `.deb` with both binaries, the bootstrap manifest (as a conffile), the systemd unit, and autogenerated dependencies (`$auto` expanded to `libc6`, `libssl3t64`, with `systemd, ca-certificates` added explicitly).

**What the run exposed.**
- **Release-build breakage.** `cargo build --release -p agentd` failed on the VM. `CapabilityRegistry::inspect` is gated `#[cfg(any(test, debug_assertions))]` so it's absent in release builds, but two production callers (`revoke_capability` in `aaos-runtime`, spawn narrowing in `agentd`) were using it. Local `cargo build` (debug) had been green for weeks; the first release build surfaced it. Fix: `token_id_of(handle) -> Option<Uuid>` always compiled, returns only the id — preserves the no-token-leak invariant. Both callers switched.
- **Socket path.** Previously `/var/run/agentd.sock` (legacy hardcode). `RuntimeDirectory=agentd` in the systemd unit expects `/run/agentd/*`; mismatch meant the unit's cleanup semantics were dead letters. Moved to `/run/agentd/agentd.sock` in `main.rs` + `config.rs` defaults. `/var/run` is a symlink to `/run` on modern Debian so external callers using the old path still reach the socket during a transition window.
- **StateDirectory vs postinst.** First draft of `postinst` created `/var/lib/aaos/` directly. Copilot review flagged it as duplicate lifecycle: systemd's `StateDirectory=aaos` already creates and chowns the dir on service start. Dropped the `postinst` shell to just user creation; state dirs come from systemd only. `non-systemd install mode is explicitly out of scope.
- **Dependency hygiene.** Initial plan listed `curl, jq` as hard `.deb` deps. Also Copilot: those are tool-wrapper concerns, not daemon concerns — belong in a future `aaos-wrappers-core` package. Cut to `systemd, ca-certificates`.
- **`ProtectSystem=strict` too aggressive.** Can't validate every runtime write path the daemon and workers may touch without doing the full audit that belongs in Phase F-b's Landlock work. Relaxed to `ProtectSystem=full` for F-a; tighten to `strict` in F-b.
- **`postrm` swallowing errors.** Initial draft used `deluser || true` / `delgroup || true`. Copilot flagged silent-success-shaped cleanup as wrong tradeoff for purge. Removed the `|| true`; the `getent` guards still handle the "already absent" case, but other failures surface.

**What shipped.**
- `packaging/agentd.service` — unit with `StateDirectory=aaos` (0750), `RuntimeDirectory=agentd` (0750), user `aaos`, `Restart=on-failure`, basic hardening (NoNewPrivileges, ProtectSystem=full, ProtectHome, PrivateTmp, ProtectKernelTunables/Modules/ControlGroups).
- `packaging/debian/postinst` — creates the `aaos` system user via `adduser --system --group --no-create-home --shell /usr/sbin/nologin`. That's it.
- `packaging/debian/postrm` — on `purge`: `rm -rf /var/lib/aaos`, then `deluser`/`delgroup` with `getent` guards but no error swallowing.
- `[package.metadata.deb]` in `crates/agentd/Cargo.toml` — `cargo-deb` config.
- One-line socket path changes in `main.rs` + `config.rs`.
- `token_id_of()` on `CapabilityRegistry` plus two caller swaps (`aaos-runtime`, `agentd`).

**Verification log on the Debian 13 VM.**
- `cargo deb -p agentd` → `target/debian/aaos_0.0.0-1_amd64.deb` (3.4MB).
- `dpkg -I` output: `Depends: systemd, libc6 (>= 2.34), libssl3t64 (>= 3.0.0), ca-certificates`. Section admin, priority optional.
- `apt install ./aaos_0.0.0-1_amd64.deb` → unit installed and disabled (by design); user `aaos` exists (uid 104).
- `systemctl start agentd` → active, 3.6 MB peak memory, socket at `/run/agentd/agentd.sock` with perms `aaos:aaos srwxr-xr-x`.
- `/var/lib/aaos/` created by systemd at start (0750 aaos:aaos, empty).
- JSON-RPC wire check: socket returns a proper JSON-RPC parse error for a malformed request (confirms the listener is serving, not just binding).
- `apt purge aaos` → user removed, `/var/lib/aaos` removed, binaries removed. Only `/lib/systemd/system` left behind (non-empty warning from dpkg, harmless).

**Cost.** ~$0.40 VM-hours on a $0.029/hr droplet while building + testing. No LLM cost (manual build; no runs triggered).

**What stays open for F-b.** Packer pipeline starting from an upstream Debian 13 image, preinstalling the `.deb`, flipping `AAOS_DEFAULT_BACKEND=namespaced`, baking Landlock + seccomp profiles, stripping desktop meta-packages, custom motd. Separate plan.

**CI.** Skipped for now. When the first real release is cut, add a GitHub Actions job that builds in a `debian:13` container and attaches the `.deb` to the release. Today's build-on-VM flow is fine while there are no users to receive an artifact.

**Lessons for patterns.md (if generalized).**
- **Release builds need their own signal.** `#[cfg(debug_assertions)]`-gated APIs that production code calls are a silent trap. `cargo build` in CI is always debug-mode; packaging flow was the first release build and caught it. Candidate pattern: at least one CI job should build with `--release`.
- **Let the platform own the platform concern.** The first draft of `postinst` re-implemented what `StateDirectory=` already does. When a systemd directive exists for your lifecycle need, use it — don't parallel-implement in shell.
