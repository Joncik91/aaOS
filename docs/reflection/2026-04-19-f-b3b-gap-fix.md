# Phase F-b sub-project 3b — closing the two Gap-3 QA findings *(2026-04-19)*

**Integration commits under test:** `0716b1a`..`9c33e4d` (5 commits on `main` after the 2026-04-19-f-b3-e2e-qa.md reflection). Closes Gap A (inline subtask → worker session) and Gap B (tool-layer capability re-check inside worker) from the prior day's droplet QA.

## Setup

Same droplet as the prior F-b/3 e2e — `159.89.31.179`, Debian 13, kernel 6.12.43. Two runs: default (`AAOS_CONFINE_SUBTASKS` unset) with the canonical HN+lobsters goal, and flag-on with a small `file_read` goal to prove workers actually run the tool calls.

## What shipped

### Gap B — capability forwarding across the broker (Task T15, commits `0716b1a..23400b5`)

- `CapabilityRegistry::resolve_tokens(&[CapabilityHandle], AgentId) -> Vec<CapabilityToken>` added.
- `Request::InvokeTool` gained `capability_tokens: Vec<CapabilityToken>` (serde-defaulted so older workers stay compatible).
- `WorkerHandle::invoke_over_worker` signature now takes `Vec<CapabilityToken>`.
- Daemon resolves the agent's handles to full token structs before sending.
- Worker builds a per-call `CapabilityRegistry`, reinserts the tokens, hands handles into `InvocationContext`.
- Tools' internal `ctx.capability_registry.permits(...)` check now succeeds against forwarded tokens — positive worker-side tool calls work end-to-end.

Droplet verification: `cargo test -p aaos-backend-linux --lib invoke_tool -- --ignored` runs both `invoke_tool_file_read_roundtrips` (positive case) and `invoke_tool_landlock_denial` (negative case) — both pass.

### Gap A — worker session for inline subtask agents (Task T16, commits `686a583..8f02b32`)

- `Server::launch_worker_session_for_subtask` added — calls `NamespacedBackend::launch` for each subtask agent, returns a scopeguard that stops the worker on return.
- `run_subtask_inline` invokes the helper; `NoSession` fallback kicks in if launch fails.
- `NamespacedBackendConfig::default()` now uses `/run/agentd/sessions` (within systemd's `RuntimeDirectory=agentd`, owned `aaos:aaos` 0750) so the packaged daemon-as-aaos-user can create it. Previously default was `/var/run/aaos/sessions` — unwritable by non-root.

### The third finding that emerged from testing Gap A

**Workers have no workspace access.** `PolicyDescription { scratch: /scratch, shared_libs, broker_socket }` gives the worker a mount namespace where only the session tmpfs and shared libraries exist. Paths like `/var/lib/aaos/workspace/<run-id>/` — where the canonical workflow's fetcher writes and analyzer/writer read — do not exist inside the worker. Confining every plan-executor subtask therefore breaks the workflow: analyzer tries `file_read` on `/var/lib/aaos/workspace/<run>/hn.html`, gets ENOENT from inside the sandbox, tells the planner it failed, replans, fails again.

Evidence from the droplet run with `AAOS_CONFINE_SUBTASKS=1`:
```
tool: file_read [worker] {"path":"/var/lib/aaos/workspace/3ef8df33-.../hn.html"}
tool FAILED: file_read — ipc error: tool error: ipc error: file not found: No such file or directory (os error 2)
```

The worker launched, the broker round-trip worked, capability forwarding worked — but the file simply wasn't visible to the process inside the mount namespace. Landlock wasn't denying; the filesystem was empty because the bind mounts don't include the workspace path.

**Design choice.** Three options considered:
- (a) Bind-mount the per-run workspace path into the worker. Needs `AgentLaunchSpec.workspace_path` to flow into `PolicyDescription` + a mount step in `clone_and_launch_worker`. Moderate scope.
- (b) Add workspace paths to the Landlock allow list + keep the host-visible mount. Simpler but erases the mount-namespace isolation benefit.
- (c) Keep the current design and make subtask confinement opt-in until (a) is designed + shipped.

v1 ships (c) — `AAOS_CONFINE_SUBTASKS=1` is opt-in, default stays on the working daemon-side path. Production workflows work; operators who want to exercise the worker path for security testing flip the flag and use tasks that only touch `/scratch` or shared libs. Closing the gap ((a) above) is its own sub-project.

## What worked

- Default canonical goal: `/data/compare.md` written at 5.7 KB, 9 daemon-side tool calls, zero panics, 182s completion.
- `AAOS_CONFINE_SUBTASKS=1` + small `file_read` goal: 18 `[worker]` tags, 0 `[daemon]` tags — every tool call routed worker-side. Workers launched cleanly (no launch warnings). Capability tokens forwarded; tool's own permits-check succeeded; then the actual behaviour (Landlock denial on `/etc/shadow`, or capability-grant denial on `/proc/self/status`) took effect.
- Audit surface honest: when a worker launch *would* have happened but the agent's workflow doesn't make sense under confinement, the NoSession fallback labels the call `[daemon]` so operators see the truth.

## What the run exposed

- The workspace-mount gap named above. Named honestly in architecture.md.
- `NamespacedBackendConfig::default()` shipped with a path (`/var/run/aaos/sessions`) that agentd-as-aaos-user cannot create. Caught only when T16 tried to use it in production — the existing integration tests override the path via a tempdir, so the misconfiguration was invisible to CI.

## Cost

Under $0.05 droplet time. One DeepSeek key rotated per run; final key shredded + droplet pending destruction.

## Lessons worth lifting

**"Named the opt-in flag honestly."** Previous pattern was "fall back quietly when the intended path fails." This sub-project does the opposite: if the confined path fails, DON'T fall back silently — require the operator to flip a flag to opt into the confined path. That makes the default behaviour predictable (daemon-side, workflows work) and makes confinement visible in the log (if you see `[worker]` tags in production, you explicitly asked for them). Candidate for patterns.md if this pattern re-appears in later sub-projects.

**Test defaults should match production configuration.** The ignored-by-default integration tests passed on CI because they override `session_dir` to a tempdir. The real `NamespacedBackendConfig::default()` was broken for 6+ months and only surfaced when the packaged daemon actually tried to use it. Package-time configuration ≠ test-time configuration — add a smoke test that runs the backend with its real default config under an unprivileged uid.
