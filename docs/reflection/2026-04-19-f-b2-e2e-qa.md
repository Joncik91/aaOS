## Setup

- **Droplet:** fresh DigitalOcean `debian-s-4vcpu-8gb-fra1`, Debian 13 (trixie), kernel 6.12.43, destroyed after the run.
- **Build:** `./packaging/build-deb.sh --features mcp --no-default-features` on droplet — the one-step wrapper shipped in `b24139b` as BUG #1's real fix. Produced a 4.5 MB `.deb` in ~4 min on a 4vCPU droplet.
- **Install:** `apt install target/debian/aaos_0.0.0-1_amd64.deb`. Daemon auto-started + auto-enabled on install — confirmation that `ce4582a`'s postinst systemd wiring still works on a clean host.
- **Key:** DeepSeek key at `/etc/default/aaos` mode 0600; shredded on exit.
- **Trigger:** `AAOS_DEFAULT_TASK_TTL_WALL_CLOCK_S=5` + the canonical "fetch HN and lobste.rs, compare the top 3 stories on each, write a detailed 800-word comparison" goal. The 5s TTL reliably forces the analyzer and writer subtasks (which have `model_ladder: [deepseek-chat, deepseek-reasoner]`) to fail at tier 0 — a `SubtaskCompleted{success:false}` signal maps to `ReplanRetry` and triggers escalation.
- **Workflow:** install → key → restart → submit → read stream → verify.

## What worked

Every acceptance criterion from the plan held, first try:

- **BUG #1 stays fixed on a fresh host.** `./packaging/build-deb.sh --features mcp --no-default-features` produced the `.deb` in one command — the worker binary was built as part of the script, not by hand.
- **BUG #2 stays fixed on a fresh host.** `apt install` left `systemctl is-active agentd` = `active` + `is-enabled` = `enabled`. Socket at `/run/agentd/agentd.sock` mode 0660 `aaos:aaos`.
- **Escalation fires end-to-end.** The stream shows the exact shape the spec promised:
  ```
  [05:32:14] 00000000    TTL expired (wall_clock_exceeded): analyze_comparison
  [05:32:14] 00000000    model escalated (replan_retry): analyze_comparison — deepseek-chat → deepseek-reasoner
  ```
  `SubtaskTtlExpired{wall_clock_exceeded}` fires → batch-failure path populates `FailedSubtask{observed_signals: [ReplanRetry]}` (via the `SubtaskCompleted{success:false}` emitted by the batch loop) → `decide_escalation` picks `ReplanRetry` → tier 0→1 → `SubtaskModelEscalated` event → operator sees it formatted in cyan in the default CLI view.
- **Per-failed-subtask escalation is independent.** The run showed three separate `model escalated` events — two for `analyze_comparison` (one per replan attempt up to ladder top at tier 1) and one for `write_report`. `carry_tiers_forward` merges tier bumps into replanned plans by subtask-id match; each subtask's tier advances independently.
- **Ladder-top cap holds.** After the second `analyze_comparison` replan, the tier was already at 1 (ladder length = 2). The third replan showed `TTL expired` again but **no matching `model escalated`** — the guardrail test `subtask_escalation_caps_at_ladder_top` from unit tests held true under real traffic.
- **Audit cleanliness.** `grep -c 'model escalated' /tmp/submit.log` = 3; `grep -c 'TTL expired' /tmp/submit.log` = 4 (the fourth is the ladder-top no-escalation case). `journalctl -u agentd | grep -iE 'panic|backtrace'` — empty. No regressions.

## What the run exposed

**Nothing.** First first-on-production exercise after a plan-level change that held up on initial try — rare. Worth noting two things that could have gone wrong and didn't:

- The `ExecutorError::Correctable { reason, failures }` struct-form refactor in T8 touched 8 construction sites + 6 pattern-match sites. Sub-project 1's re-verification flagged two shipped bug-fixes that actually didn't work on a fresh host; this sub-project's analogous risk (the enum-to-struct refactor having subtle call-site bugs) did not materialize.
- The `CompositeLatencyTracker` — wrapping both trackers via `Vec<Arc<dyn LatencyTracker>>` — had the potential to mis-route on `SchedulerView::new` (register the subtask under the wrong model name, or register twice on re-wrap). The manifest-parses-cleanly path held: every subtask's `&manifest.model` was the tier-selected model, so the per-model tracker saw the right bucket.

## Lessons worth lifting

**One-fix-two-bugs was a sub-project-1 failure mode.** Sub-project 2's e2e passed first-try because of three specific process decisions carried forward:

1. **Spec named the operator-visibility requirement as a first-class task** (T10), not a decoration step after implementation. Sub-project 1's `SubtaskTtlExpired` shipped invisible to operators by default; sub-project 2's `SubtaskModelEscalated` was visible from day one.
2. **Fresh-droplet verification was Task 14, not an afterthought.** Sub-project 1 needed a second reflection entry (`2026-04-18-f-b1-bugfix-reverify.md`) after two shipped fixes turned out broken on a real host. Sub-project 2 treated on-host as part of Definition of Done; it ran in one shot with no corrections.
3. **The spec's "scope notes" were honest.** Gap 1's original description oversold what `SchedulerView` wrapped; BUG #4 was a docs-honesty fix. Sub-project 2's spec said up front that `PerModelLatencyTracker` is observability-only, not consumed by v1 routing. No retroactive doc correction needed.

These three are already in `patterns.md` from sub-project 1's reflection; no new pattern to lift. What's new is evidence that applying them prevents the re-verification regressions they're meant to prevent.
