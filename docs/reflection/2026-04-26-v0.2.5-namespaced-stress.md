# 2026-04-26 — Namespaced-backend stress probe on v0.2.5

Companion to the v0.2.5 stress probe (which ran under InProcessBackend and found Bug 35).  Re-running the same harness with `AAOS_DEFAULT_BACKEND=namespaced` exercises the broker `SessionMap` + worker fork/exec/landlock/seccomp setup path that the InProcess run didn't touch.

**Two real bugs surfaced — both shipped as v0.2.6.**

## Setup

- Same droplet as the v0.2.5 stress run.
- v0.2.5 `.deb`, `AAOS_DEFAULT_BACKEND=namespaced` set in `/etc/default/aaos`.
- `AAOS_NAMESPACED_CHILD_DEBUG=/var/log/agentd/child-debug.log` enabled to capture the worker's setup-step log on failure.
- Stress harness updated: `lifecycle: bootstrap` → `lifecycle: persistent` so spawned agents actually go through `backend.launch` (the on-demand path skips `launch` entirely — the v0.2.5 InProcess run was hitting only the registry/capability shard locks, not the namespaced fork+exec).

## What it found

### Bug 36 — `mount("proc", ..., "proc", ...)` fails inside unprivileged user namespace

First spawn under namespaced+persistent returned *"worker exited with status 72 before sandboxed-ready."*  Status 72 = Step E2 (the procfs mount the v0.2.1 TOCTOU fix added).  Child debug log:

```
A-pipe-read: ok
B-ms-private: ok
C-tmpfs-newroot: ok
D-scratch-tmpfs: ok
E-bind-/lib/x86_64-linux-gnu: ok
E-bind-/lib64: ok
E-bind-/usr/lib/x86_64-linux-gnu: ok
E2-proc-mount: ERR EPERM: Operation not permitted
```

Root cause: `mount("proc", path, "proc", flags)` inside an unprivileged user namespace requires the namespace to also own a PID namespace.  The worker deliberately *doesn't* unshare `CLONE_NEWPID` (the module doc on `lib.rs` line 14 explicitly says so — avoids PID-1-in-namespace semantics).  v0.2.1's procfs-mount fix conflicted with v0.0.x's choice to skip CLONE_NEWPID and didn't trigger anywhere in v0.2.1–v0.2.5 testing because:

- Every droplet QA run the canonical fetch-HN goal.
- The canonical goal goes through `agent.submit` → plan executor → `execute_agent_for_subtask` (inline, never calls `backend.launch`).
- So the only way to hit `backend.launch` for a persistent agent is via JSON-RPC `agent.spawn` directly + `lifecycle: persistent` + namespaced — not a path the canonical QA exercises.

The Bootstrap-via-Docker entrypoint *does* spawn-as-persistent + launch through the backend, but Docker doesn't set `AAOS_DEFAULT_BACKEND=namespaced` so it gets InProcess.  The systemd `.deb`'s `agentd configure` doesn't copy the namespaced setting from the `.example` (Bug-not-numbered: the README claim that .deb enables namespaced-by-default; documented and corrected in commit `9d2a399`).  So the practical universe of operators hitting Bug 36 was: anyone who manually sets `AAOS_DEFAULT_BACKEND=namespaced` AND uses JSON-RPC `agent.spawn` directly with `lifecycle: persistent`.  Empty set in current usage, but a real defect.

**Fix**: bind-mount host `/proc` instead of mounting fresh procfs.  Bind-mount needs no userns privilege.  `/proc/self` is a magic-link resolved per-task by the kernel — the worker's own `/proc/<host-pid>/*` is reachable regardless of which procfs instance is mounted, so `readlinkat("/proc/self/fd/<fd>")` works.  Trade-off: worker can also see other host PIDs, but Landlock denies any read outside the explicit allow-list (which doesn't include `/proc/<other>`), so visibility is recoverable only as a side channel, not direct exfiltration.  Acceptable.

### Bug 37 — `agent.stop` leaks worker subprocess under namespaced+persistent

After Bug 36 was fixed and the smoke spawn succeeded, the post-test check showed: *20 spawns, 20 stops, 0 errors, 20 `aaos-agent-worker` processes still alive.*  Status `S` (sleeping).  None of them zombies — they were *running*, waiting for the next message in their broker-read loop.

Root cause: `AgentRegistry::stop` ends the in-daemon persistent loop (sends `AgentCommand::Stop` to the channel, awaits the task handle) but never tells the namespaced backend to terminate the *subprocess*.  `NamespacedBackend::stop` exists and correctly does SIGTERM + waitpid, but it was only called from tests.  Production `Server::handle_agent_stop` only called `registry.stop`.

The worker subprocess detects broker-stream EOF and exits cleanly — but the daemon-side `BrokerSession` was never dropped, so EOF was never sent to the worker, so the worker just sat in its read loop forever.

**Fix**: new `AgentBackend::stop_by_agent_id` trait method (default no-op for backends without subprocesses; `NamespacedBackend` overrides to look up its session by agent_id and SIGTERM+waitpid the worker pid stored on the session).  `Server::handle_agent_stop` calls it after `registry.stop` succeeds.  Best-effort — errors are logged but don't propagate, since `registry.stop` already succeeded and the operator-visible state is correct.

## Verification

3-pass leak test on v0.2.6 droplet (16 workers × 100 spawns × 3 passes = 4800 namespaced spawn-stop cycles on the same daemon):

| Pass | RSS before | RSS after | Δ | Wall clock |
|---|---|---|---|---|
| 1 | 20.2 MB | 29.3 MB | +9.1 MB *(initial allocations + Bug 35 cap fill)* | ~24 s |
| 2 | 29.3 MB | 31.9 MB | +2.6 MB | ~24 s |
| 3 | 31.9 MB | 32.5 MB | +0.6 MB | ~24 s |

Memory stable; passes 2-3 are essentially flat.  Throughput: ~67 spawn-stop/sec under namespaced (vs ~160/sec under InProcess from the v0.2.5 run).  6.6× slower, expected — fork+exec+namespace+landlock+seccomp setup is real overhead.

Worker process count: 0 zombies, 0 leaks across 4800 cycles.

## Pattern lifted

> **"Test the path the canonical goal actually uses" misses bugs the canonical goal doesn't use.**
>
> The canonical fetch-HN goal goes through inline plan-executor subtasks (no `backend.launch`).  Every droplet QA since v0.0.2 has run the canonical goal end-to-end and concluded "namespaced backend works."  But namespaced+persistent+`agent.spawn` is a real production path (any operator using JSON-RPC directly hits it), and that path was broken from v0.2.1 (Bug 36) and v0.0.x (Bug 37).
>
> The canonical-goal QA discipline catches *regressions to the canonical path*.  It doesn't catch *bugs in surfaces the canonical path doesn't touch*.  Stress probes that exercise every JSON-RPC method surface those.  Wiring `stress-droplet.sh` (under both InProcess and namespaced) into the release checklist alongside the canonical run would catch this class going forward.

## Cost

- Probe setup time: ~30 s (smoke test under namespaced revealed Bug 36).
- Bug 36 diagnosis: ~5 min (read child debug log + grep for exit code 72 source + reason about CLONE_NEWPID requirement).
- Bug 36 fix: ~10 min (bind-mount instead of fresh procfs + verify on droplet).
- Bug 37 diagnosis: ~3 min (post-test check showed 20 alive workers; grep for `backend.stop` showed no production callers).
- Bug 37 fix: ~15 min (trait method + impl + wire into Server).
- Verification: ~5 min (3-pass leak test).
- Total: ~70 min from "design probe" to "v0.2.6 release commit ready."

## Follow-ups

1. **Wire stress-droplet.sh into the release checklist.**  Both `STRESS_WORKERS=4 STRESS_SPAWNS=20` (smoke) and `16×100` (full) under both InProcess and namespaced.  Catches Bug-35/36/37-class regressions before release.
2. **Audit other `backend.*` methods for "called only from tests" patterns.**  `health()` is the obvious next candidate — if the daemon never calls it, the supervisor restart-on-failure logic is dead code.  Quick grep + verify.
3. **Bug-37-class prevention going forward.**  The pattern is "trait method exists, no production caller" — `backend.stop` was tested but unused.  Consider adding a `#[must_call_in_production]` lint or a release-checklist item: every `AgentBackend` method must have at least one non-test caller.  Cheap to enforce; closes a class of forgotten-cleanup bugs.
