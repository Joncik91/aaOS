## What the run exposed

### Bug 28 — `web_fetch` redirect host bypass (high)

`crates/aaos-tools/src/web_fetch.rs` constructed its `reqwest::Client` with `Policy::limited(5)`, so up to five HTTP redirects were followed automatically without re-checking the destination host against the agent's `NetworkAccess` capability grant.  The capability check at the top of `invoke()` ran exactly once — on the URL the agent originally requested.  An attacker who controlled a server inside the agent's allowed host list (or compromised a service the agent legitimately interacts with) could return a `302 Found` with `Location: https://attacker-controlled/steal`; reqwest would silently follow, the agent saw the attacker's response body as the tool result, and the audit log recorded only the original permitted URL.

**Fix shape**: build the client with `Policy::none()` so reqwest stops following redirects.  Hoist the capability check into a free function `check_url_permitted(ctx, url)`.  Wrap the GET in a manual loop: check, send, if 3xx + Location then re-resolve (handle relative URLs against the current URL) and loop with re-check.  Hop counter caps at `MAX_REDIRECTS`.  Test added — `redirect_to_unpermitted_host_denied` spawns a mock TCP server that returns a 302 to `attacker.example.com`, asserts `CapabilityDenied`.  Commit `eca9ddb`.

### Bug 29 — `file_list` residual TOCTOU on directory listing (medium)

`crates/aaos-tools/src/file_list.rs` correctly opened the requested path with `O_PATH | O_NOFOLLOW` to derive a kernel-pinned canonical via `/proc/self/fd/<fd>` and run the capability check on it — but **immediately dropped the fd** (`let (_fd, c) = ...`) before the actual I/O.  The subsequent `tokio::fs::metadata(path)` and `tokio::fs::read_dir(path)` calls re-resolved the path string from scratch.  Between the fd drop and the second open, an attacker with write access to any path component could swap the directory for a symlink to a forbidden tree (e.g. `/etc`); the capability check passed against the original inode, but the listing was performed against the swapped target.  The code's own comment acknowledged this on lines 46-48 ("For full immunity we'd need fdopendir on the same fd...") — but the deferred state meant the fix wasn't actually shipped, and the bypass was real.

**Fix shape**: new `AccessMode::ReadDir` in `path_safe` (`O_RDONLY|O_DIRECTORY|O_NOFOLLOW|O_CLOEXEC`).  `file_list` rewrite tries ReadDir first; on `ENOTDIR` falls through to Read mode for the single-file branch.  Either way, the fd survives the capability check and powers the actual I/O — `fstat` for files, `nix::dir::Dir::from_fd` + iter for directories.  No second open by path string; a leaf-component symlink swap between check and listing has no effect.  Side effect: directory-listing entries now report `size_bytes: 0` because per-entry `fstatat` by name would re-introduce the TOCTOU surface we just closed.  Operators who need entry sizes can `file_read(path)` the entry of interest.  Commit `6b24cf7`.

### Bug 30 — non-atomic session-store rewrite can permanently destroy history (high)

`crates/aaos-runtime/src/persistent.rs` summarization path called `session_store.clear(&agent_id)` (truncating the JSONL file to zero) followed by `session_store.append(&agent_id, &history)` (rewriting the now-much-shorter history).  Steps 3 and 4 were not atomic — and the code's own comment on lines 97-101 acknowledged this as a "deferred follow-up".  If the daemon crashed between the clear (file truncated) and the append (rewriting), the on-disk file was empty.  In-memory history was still intact and would be rewritten the *next* summarization cycle, but a daemon restart in between loaded the empty file.  The agent's session history was permanently destroyed.  Filesystem-full at the wrong moment had the same effect.  The 60 s throttle on `SessionStoreError` audit events meant operators got at most one warning per minute even on persistent failures.

**Fix shape**: `SessionStore::replace(&agent_id, &messages)` becomes a trait primitive with a default `clear+append` fallback (in-memory stores don't need atomicity).  `JsonlSessionStore` overrides replace with the standard write-temp + fsync + `rename(2)` pattern.  POSIX rename is atomic on the same filesystem so readers always see either the old or the new history, never an intermediate empty file.  Temp file cleaned up on rename failure.  `persistent_agent_loop` calls replace once instead of clear-then-append.  Test added — `jsonl_replace_is_atomic_swap` seeds 3 messages, replaces with 2, asserts the load returns the 2 new messages and no `.tmp` file is leaked.  Commit `4bdfb5b`.

## Process notes

- **The deferred-follow-up paper trail is itself a finding.**  Bugs 29 and 30 both came from comments in v0.2.x code that flagged a known limitation as "deferred follow-up".  Either the comments are correct *and* the bugs are real (fix them, this round), or they're stale paperwork (delete them).  Either way the comments are signal: if a code comment says "this is deferred", the next reflection-loop pass will probably file it as a finding.  Don't ship deferred comments without an `ideas.md` entry naming the reconsider signal — make the deferral structural, not buried in code.
- **The "v0.2.x is recent — fresh eyes" hint paid off.**  Three real findings out of three is the highest hit rate since round 1.  Whether this generalizes is unclear: rounds 2-5 used a more open-ended "find up to 3 bugs in the source" prompt and the noise floor climbed steadily.  This round's narrower prompt reads more like an audit-the-deltas pass than a fresh eyes pass.  Worth trying again for v0.2.2 — does the loop find new ground in v0.2.2 deltas, or does the rate drop to round-5 levels?
- **Triage time was minimal.**  Each finding was confirmed by one targeted source read (file_path:line), comparing the agent's claim against the actual code.  Total triage: ~5 minutes for all three.  The reproduction sketches were concrete enough that no separate "is this really a bug" judgment call was needed.
