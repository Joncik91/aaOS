# 2026-04-26 — Round 11 self-reflection on v0.2.7 (silent-failure prompt shape)

Round 11 of the self-reflection loop.  Round 9 with the standard prompt: 0/3 real findings on v0.2.4 → declared "depleted."  Round 10 with the trait-method-with-only-test-callers prompt: 2/3 real on v0.2.6 (Bugs 38, 40).  Round 11 with the silent-failure prompt: 3/3 real on v0.2.7 (Bugs 41, 42, 43).

**The "depletion" pattern is consistently per-prompt, not per-runtime.  Three distinct prompt shapes have surfaced 5 real bugs across rounds 10-11.**

## Setup

- Same droplet as rounds 5–10.
- v0.2.7 `.deb`; v0.2.7 source cloned from the GitHub `v0.2.7` tag.
- `AAOS_DEFAULT_BACKEND` unset (rounds-1-10 protocol).
- Goal text: explicit prompt-shape steer toward `let _ = result_returning_call()` discards, with the requirement to classify ALL discards (not just find some) so the maintainer can verify coverage.  Full prompt in commit message of the round-11 submission.
- 205 s wall clock from submit to "report complete" — slowest v0.2.x round to date because the prompt asks for an exhaustive sweep + classification, not just up to 3 findings.  ~17 KB report.

## What worked

- **Asking for the classification table changed the shape of the report.**  Round 10's 11KB report had 3 findings + summary.  Round 11's 17KB report has 3 findings + a 17-row table classifying every other `let _ = ` discard with reasons.  The table itself is useful as a future maintenance reference — next time someone modifies one of those sites and wonders if the discard is safe, the table answers it.  Convention going forward: when steering toward a finding shape, also ask for the *non-findings* (sites that match the shape but are defensibly discarded).  The maintainer can audit completeness, and the agent is forced to read every match rather than stop at the first interesting one.
- **All three findings are persistence-layer silent failures**, all with the same fix shape (audit event + log).  That's a coherent class — the runtime's "best-effort" persistence has been degrading silently across an entire release line.  The fix surface is small: thread the existing `SessionStoreError` audit event kind through three more sites.  No architectural change required.
- **The "should_emit" throttle pattern from the v0.2.2 Bug 30 fix** was the right primitive to extend.  Bugs 41 and 42 reuse the existing `last_session_store_error` Instant variable + `SESSION_STORE_ERROR_THROTTLE` constant directly.  Adding a new audit event kind would have been more invasive; "extend the existing one to cover three more operations" is a one-line change per site.

## What the run exposed

### Bug 41 — `archive_segment` failure permanently destroys conversation history

`persistent.rs:98` (v0.2.7) discarded `archive_segment` errors via `let _ = `.  The next two lines unconditionally drained the archived messages from in-memory history.  If the archive write failed, the original messages existed nowhere — gone from memory, not on disk.  Daemon restart loaded only the LLM-summarized form.  No audit, no log.

The agent's POC was concrete: `dd if=/dev/zero of=/var/lib/aaos/sessions/.fill bs=1M` to fill the partition, then trigger summarization, observe the data loss after restart.  Real disk-full or NFS-outage scenarios produce the same effect during normal operation — no attacker required.

**Fix shape**: skip the summarization cycle when archive fails (preserve original history; let the next cycle retry).  Audit + log on failure (with the `should_emit` throttle).  The change wraps the existing summarization branch in an `if let Err` else-branch so the success path is unchanged.

### Bug 42 — `append` failure silently truncates on-disk history

`persistent.rs:208` did `let _ = session_store.append(...)`.  Same shape as Bug 41 but on the per-turn append path, not the summarization path.  Failure means the agent's response for that turn is gone from disk forever — daemon restart loads history only up to the last successful append.  No audit, no log.

The pre-existing replace-failure handling at line 114-117 already had the throttle pattern.  Bug 42 fix extends it to cover the append site.

### Bug 43 — silent SQLite orphan rows in approval store

Three sites in `ApprovalQueue::request` (timeout-handler, oneshot-fail, timeout-elapsed) plus the daemon's startup purge in `Server::build_approval_queue` did `let _ = store.remove(id);`.  Under read-only-fs or SQLite-locked conditions the row stayed in the database while the in-memory `DashMap` entry was removed.  Subsequent `agent approval list` showed empty; the table grew.  After restart the entries reloaded, retried, silently failed again, accumulating across every daemon lifetime.

The `respond` site at line 103 already logs (commit `860491c` from v0.2.0 work).  The three other sites + the startup-purge site were oversights of the same pattern.

**Fix shape**: log at `warn!` with `approval_id` + error.  Mirrors the existing `respond` site exactly.

## What the agent classified as defensibly discarded

The full table is in the round-11 report at `/data/round11-findings.md` on the droplet.  Highlights worth preserving here:

- **Best-effort cleanup in stop paths** (`kill`, `waitpid`, `remove_file` on stale sockets): defensible because the failure modes are "child already exited" or "socket doesn't exist yet."
- **Broadcast send to no subscribers** (`broadcast_audit.rs:37`): defensible because the inner sink already received the event.
- **OnceLock setters** (`set_notifier`, `set_router`, `plan_executor.set`): defensible because the second call is a structural no-op by design.
- **NDJSON write to disconnected CLI** (~12 sites in `server.rs`): defensible because the handler already returns immediately on the first write failure.
- **Drop-based scopeguard cleanup**: defensible because tokio runtime drop will reap.
- **`let _ = expr?`**: discards the unit return after `?` propagates the error — not actually a silent discard.

The 44 defensible cases share a structure: the discard is correct when the failure mode is *idempotent* (cleanup) or *ignorable* (no subscribers, second call no-op).  The 3 buggy cases share a structure: the discard hides a *real state divergence* between memory and disk where the operator-visible consequence accumulates over time.

Three rounds in a row (9 → 10 → 11), three different prompt shapes:
- Round 9 (standard): 0 real
- Round 10 (trait-with-only-test-callers): 2 real
- Round 11 (silent-failure-discards): 3 real

Don't declare structural depletion until 3-4 different shapes all return zero.

Candidate shape steers for round 12:

- "Find `Arc<dyn Trait>` fields where the trait has methods nobody calls" (Bugs 35/37/38 generalized)

- "Find atomic CAS loops that don't bound retry count" (livelock class)
- "Find regex compilations inside hot loops" (perf class — different surface)

## Process notes

- **Asking for the non-findings is the correct move.**  Round 11's "classify all 47 sites" requirement caught the moment the agent might have stopped at "I found 3, that's enough."  The classification table is itself a deliverable — a reviewer can audit "did you actually look at all 47?" by spot-checking entries.
- **Prompt-shape exhaustion is a cheap way to extend the loop's reach.**  Each new prompt shape costs ~5 min runtime + ~5 min triage + a release commit.  At a conservative 1 real bug per prompt shape, that's a much better return than continuing to iterate the same shape and declaring depletion at zero.
- **The `SessionStoreError` audit event kind is now load-bearing.**  Originally added for the v0.2.2 Bug 30 `replace` failure path; round 11 extends it to `archive` and `append`.  Documenting it as the canonical signal for any session-store I/O failure would be worth doing — operators reading the audit stream should see all three operation kinds (`replace`, `archive`, `append`) and know they're all the same class of issue.

## Follow-ups

1. **Round 12 with another prompt shape.**  Pick one of the candidate steers above.  The runtime is still producing real bugs at one new shape per round; cheap to keep going.
2. **Document `SessionStoreError` operation taxonomy in `docs/architecture.md`** — the audit event kind now covers four operations (`replace`, `archive`, `append`, plus future), each meaning a different kind of state divergence.  Worth a one-paragraph reference for operators reading the audit stream.
3. **Stress probe with disk-full simulation.**  Bug 41/42 fixes are correct in code review but the actual behavior under ENOSPC (does the daemon stay responsive? does the audit event reach subscribers? does the persistent loop continue?) is untested.  A docker-based test that mounts a tmpfs with a fixed cap and triggers summarization would close that loop.  Not blocking; queueing as a future-test task.
