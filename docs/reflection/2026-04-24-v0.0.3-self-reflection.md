# 2026-04-24 — v0.0.3 self-reflection, Bug 8 found and shipped

First self-reflection run against a tagged release `.deb` under full confinement.  aaOS reads its own source tree, reports bugs, the operator ships the fix and re-releases — the loop the project exists to run.

## Integration commits

- `aaf82a3` fix: route grep daemon-side under confinement (Bug 8)
- `<release>` release: bump workspace crates to 0.0.4

## Setup

- Fresh DigitalOcean droplet, Debian 13 (trixie), kernel 6.12.43+deb13-amd64, 2 vCPU / 3.8 GiB RAM.
- `apt install ./aaos_0.0.3-1_amd64.deb` straight from the GitHub release artifact.
- `/etc/default/aaos` mode 0600 root:root with `DEEPSEEK_API_KEY=<key>`, `AAOS_DEFAULT_BACKEND=namespaced`, `AAOS_CONFINE_SUBTASKS=1`.  Note: the postinst-generated `/etc/default/aaos.example` *does* include the backend vars, but `agentd configure` writes only the API-key vars — operators who use the interactive flow end up on `InProcessBackend` by default despite the probe having succeeded.  Filed as a known UX gap to fix in a later release; not part of Bug 8.
- Source tree cloned at `/src/aaOS` (git `v0.0.3` tag, commit `9654dff`) so the `generalist` role's `file_read: /src/*` capability would resolve cleanly.

## The goal

Adversarial prompt in the shape Run 9 validated a year ago:

> Find a concrete bug, design flaw, race condition, or security issue in the current implementation. Produce a specific file:line report with reproduction steps or a failing-test sketch. Do NOT re-propose items already documented in `/src/aaOS/docs/ideas.md`, `/src/aaOS/docs/roadmap.md`, or any file under `/src/aaOS/docs/reflection/` — those represent known-and-tracked work. Find something new.

Two submits, two different execution paths.

### Submit 1 — generalist (computed orchestration)

The Planner decomposed the goal into a 6-subtask DAG: `list → scan → verify → doc-check → synthesize → write`, each subtask handing off intermediate results through `/tmp/candidate_bugs.txt`, `/tmp/verified_bugs.txt`, etc.  But the `generalist` role has capabilities for `/data/*`, `/src/*`, `/output/*` — not `/tmp`.  Every intermediate read/write hit `capability denied: lacks FileRead { path_glob: "/tmp/..." }`.  The final writer subtask read an empty `/tmp/verified_bugs.txt` (another subtask had written the literal string `test` to it while probing for write access) and wrote a 166-byte report saying "contents are: test".  12 capability denials, 6 completed subtasks, 228 s wall clock, effectively $0.

**Not a new bug — a manifestation of a known gap** (Planner doesn't know what paths each role can touch).  Treated as a setup failure.

### Submit 2 — reflector (custom role, single-agent)

Wrote a purpose-built `reflector` role at `/etc/aaos/roles/reflector.yaml`:

- Capabilities: `file_read` + `file_list` + `file_read_many` + `grep` over `{target_dir}/**`, `file_write` to `{output}` only.
- Parameters: `target_dir` + `output`.
- Model: `deepseek-chat` with `deepseek-reasoner` as tier-1 escalation.
- System prompt: single-agent procedure (list → choose files → read → grep to verify → write one report).  Hard rule: "do all investigation in your own context, no scratch files."
- Budget: 200k input / 8k output, 2 retry attempts.

Submitted a goal that explicitly named the role.  The Planner picked it, produced a single-subtask plan, spawned one reflector.

**The reflector did 34 tool calls in 45 seconds of useful work:**

- `file_list` every crate and its `src/` directory.
- `file_read` on 15 files in the security + tool-routing boundary: `capability.rs`, `capability_registry.rs`, `tool_surface.rs`, `backend.rs`, `manifest.rs`, `audit.rs`, `backend-linux/src/lib.rs`, `backend-linux/src/broker_session.rs`, `aaos-tools/src/invocation.rs`, `file_read.rs`, `file_write.rs`, `context.rs`, `runtime/src/process.rs`, `runtime/src/registry.rs`.
- Then, to verify a candidate pattern across the tree, called `grep`.
- **Every `grep` call failed**: `ipc error: failed to spawn rg: Operation not permitted (os error 1)`.  Eight consecutive failures.
- Reflector returned `complete` with no output.  Planner retried with a second reflector; same failure mode.  Planner fell back to spawning a generalist which wrote a "Fallback Report: NOT COMPLETED" marker to `/data/report.md` and marked the run `complete`.

Total: 145 s wall clock, 0k tokens reported (all cache-hits), run marked successful, operator sees a degraded output file with no tool-failure signal.

## What the run exposed

### Bug 8 — `grep` non-functional under the namespaced backend

The `grep` tool shells out to `rg` (ripgrep) as a subprocess.  The worker's seccomp kill-filter denies `execve`, so every `grep` call under confinement fails with `Operation not permitted`.  Same class as Bug 7 (memory tools absent from the routing list).

Concretely:

- `crates/aaos-backend-linux/src/worker_tools.rs:24` lists `"grep"` in `WORKER_SIDE_TOOLS`.
- `crates/aaos-core/src/tool_surface.rs:31-38` does NOT list `"grep"` in `DAEMON_SIDE_TOOLS`.
- `crates/aaos-core/src/tool_surface.rs:73` has a test that *asserted* the buggy routing: `assert_eq!(route_for("grep", "namespaced"), ToolExecutionSurface::Worker)` — the bug was enshrined in the test suite.
- `crates/aaos-tools/src/grep.rs` invokes `rg` via `std::process::Command` — the call that trips seccomp.

Ironically, this is the very bug that *prevented* the reflector from finding the bug.  A self-reflection run that can't `grep` is reduced to reading whole files — possible but much slower and prone to missing cross-file patterns.

Fixed in `aaf82a3`: moved `"grep"` to `DAEMON_SIDE_TOOLS`, removed the `GrepTool` registration from `build_worker_registry`, flipped the test assertions, expanded doc comments to name three distinct daemon-side routing reasons (network, subprocess-exec, HTTP).  Shipped as `v0.0.4`.

### Bug 9 (queued) — silent fallback-report cascade on tool failure

When the reflector (or any role) returns `complete` with no output because its tools failed, the Planner's replan path spawns a fallback agent that writes a "NOT COMPLETED" marker to the output path and marks the whole run `complete`.  The original tool-failure events are in the audit log, but the CLI's end-of-run summary and the output file both present a successful-looking result.  Silent quality degradation — the most dangerous outcome in an orchestration system.

Not shipped in v0.0.4 — fix needs a design decision on how to surface partial success.  Three candidates:

1. **Propagate tool-failures into the replan signal** so the Planner's retry prompt sees the specific failure ("grep failed 8 times with execve denial") and can either route around it or fail the plan.
2. **Fail the whole plan** if any subtask returns `complete` with zero output.  Simplest; most aggressive.
3. **Write a partial report with inline errors** — still produce an output file, but prefix it with a "Tools unavailable during this run: [grep × 8]" summary block so the operator sees the cause.

Option 3 is probably the right user-facing shape, but option 1 is the cheaper and more honest runtime change.  Tracked under `CHANGELOG.md` `[Unreleased]` as Bug 9.

### Architectural watchlist — routing-list drift

Bug 7 and Bug 8 are the same bug, surfaced twice.  Two sources of truth (`WORKER_SIDE_TOOLS` and `DAEMON_SIDE_TOOLS`) can drift — and when they do, tools that look correctly-registered fail at runtime with opaque IPC errors.  The existing test suite exercises both lists *independently* but has no invariant tying them together.

A clean refactor: a single `TOOL_SURFACE_REGISTRY` as the source of truth, with `WORKER_SIDE_TOOLS` / `DAEMON_SIDE_TOOLS` derived from it at startup.  Not a blocker; added to the watchlist in `CLAUDE.md` (project-internal, not this file) so the next time someone adds a tool they consolidate rather than adding a third divergent list.

## Design / review notes

- **Purpose-built roles beat generic orchestration for focused tasks.**  Submit 1 (generalist + Planner) failed because the Planner picked unreachable workspaces.  Submit 2 (reflector, single-subtask) matched the work's shape exactly: one agent, pre-scoped capabilities, no intermediate hand-offs.  The prior Run 9 ("adversarial bug-hunt") had used the Bootstrap-Agent path for the same reason.  Pattern: **when the work doesn't need parallelism, don't use a DAG**.
- **The reflector role file is in `/etc/aaos/roles/` on the droplet only.**  It didn't ship in v0.0.4 — the question of "should aaOS bundle a reflector role by default" is a release-shape question for v0.0.5+ or M1.  For now it's a local artifact of the run.
- **Self-reflection under confinement is a different testbed than self-reflection without it.**  Prior runs (1-11) ran in Docker without Landlock/seccomp.  This run was the first where the confinement itself was on the critical path of the agent's own work — and it immediately surfaced a confinement-routing bug that the earlier environment couldn't have exposed.  **The loop closes on itself.**
