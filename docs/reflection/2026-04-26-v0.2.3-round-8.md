## What the run exposed

### Bug 34 — seccomp socket allowlist over-permissive + lying docs (medium)

`crates/aaos-backend-linux/src/seccomp_compile.rs:161-177` extended the worker's allowlist with the full AF_UNIX socket API plus server-side primitives (`SYS_bind`, `SYS_listen`, `SYS_accept4`, `SYS_accept`).  No argument filter on `SYS_socket` or `SYS_socketpair`.  Two downstream doc comments — `tool_surface.rs:26` and `worker_tools.rs:22` — both claimed "`web_fetch`: seccomp allowlist has no socket/connect syscalls".  Factually wrong.  An auditor reading the docs would conclude that a compromised worker cannot create network sockets; the code permitted exactly that.

Defence-in-depth hole, not an active exploit (no current code path lets a worker request a non-AF_UNIX socket — but capability tokens are policy, not enforcement, and the kernel-level layer here was lying about what it permitted).

**Fix**: argument-filter `SYS_socket` and `SYS_socketpair` via `SeccompCondition` so arg0 must equal `AF_UNIX` (1).  `AF_INET`/`AF_INET6`/`AF_NETLINK`/etc. now return EPERM.  Server-side primitives (`bind`/`listen`/`accept`/`accept4`) removed from the allowlist entirely — the worker is a client only.  Module-level doc + both downstream doc comments rewritten to be honest about what's allowed.  Test `seccomp_drops_server_socket_primitives` asserts the static list, `seccomp_socket_filter_compiles_with_af_unix_condition` asserts the argument-filtered allowlist compiles.

### Finding 2 — `resolve_tokens` wire race (deferred to `docs/ideas.md`)

The agent correctly identified that v0.2.0's push-revocation protocol does NOT fully close the residual race the doc-comment described.  Push-revocation closes the *post-dispatch* race for the worker's session-level registry; it does NOT close the *wire-race* window where an `InvokeTool` and a `RevokeToken` cross on the broker stream.  An attacker who calls `revoke()` after a worker has already started executing a tool with that token sees the in-flight call complete with the revoked token honored.

The proposed fix (sequence/generation counter on every token, verified at result-return) is real but heavyweight: it touches the wire format, broker schema, and audit shape, and adds latency to every tool call.  The race is microsecond-scale and requires the attacker to be inside the daemon (where the trust boundary they would defeat next is the same one that already failed).  Cost-vs-value ratio is poor at v0.2.x size.

**Action**: rewrite the `resolve_tokens` doc to be honest about what's closed and what isn't, file the heavier fix as a `docs/ideas.md` entry with two reconsider signals (multi-operator daemon needing sub-call revocation latency, or broker protocol gaining synchronous result-ack for other reasons).

### Finding 3 — `SchemaValidator` shallow type-checking (deferred to `docs/ideas.md`)

The validator does shallow type-matching only — no recursive `properties`, no `items` for arrays, no `pattern`/`enum`/`format` enforcement.  A schema like `{"type": "object", "properties": {"path": {"type": "string"}}}` accepts `{"path": 42}`.  The agent's framing (Low-Medium design flaw) was right.

Today every tool defends its own input via `input.get("path").and_then(|v| v.as_str()).ok_or(...)` patterns — type confusion produces a structured tool error, not a security bypass.  The validator is a developer ergonomic, not a trust-boundary layer.  Fix (pull in the `jsonschema` crate) is real but closes no known bug today; reconsider when externally-authored manifests need pre-tool-body schema enforcement, or when MCP integration matures to honour remote schemas locally.

**Action**: file as `docs/ideas.md` entry; no code change in v0.2.4.
