## What worked

- **Baseline computed orchestration still ships.** "fetch HN and lobste.rs, compare the top 3 stories on each, write to /data/compare.md" ran start-to-finish in 90s. Fetcher scaffold wrote `hn.html` (34 KB) + `lobsters.html` (50 KB). Analyzer read both. Writer produced a 5.3 KB `/data/compare.md` with real story titles visible on the live homepages at submission time ("Michael Rabin has died", "Migrating from DigitalOcean to Hetzner"). Zero capability denials. Same behavior as the 2026-04-17 run, so the scheduler/TTL refactor didn't regress this path.
- **MCP server end-to-end.** With `server.enabled: true` in `/etc/aaos/mcp-servers.yaml`, `agentd` exposed `127.0.0.1:3781`. `tools/call submit_goal` returned a JSON `run_id`; `get_agent_status` returned `"running"`; SSE stream at `/mcp/events?run_id=...` emitted real-time `data: {"kind":"tool_invoked",...}` frames. First real external exercise of the Phase F-a MCP server on a non-localhost environment confirms the stream wiring + loopback-only binding both behave.
- **Inference concurrency bounded.** With `AAOS_MAX_CONCURRENT_INFERENCE=1` and two parallel `agentd submit` calls, DEBUG-level permit logs (`aaos_llm::scheduled`) showed `available_permits=0` after each acquire, and acquisitions were sequential. The LEGACY `ScheduledLlmClient` is still doing the gating — see "What the run exposed" below.
- **Wall-clock TTL actually kills subtasks.** With `AAOS_DEFAULT_TASK_TTL_WALL_CLOCK_S=3`, the analyzer/writer stage of the baseline goal got killed on every iteration, the planner replanned three times, Bootstrap gave up at 53s. Behavior-visible proof that `race_deadline` fires and cancels the runner future. Not surfaced as an explicit event to the operator (see below).
- **Clean journal.** 603 lines over 25 minutes of testing. `grep -iE "panic|backtrace|unwrap.*None|index out of bounds"` returns nothing. No agent crashes; state transitions all followed the `Starting → Running → Stopping → Stopped` contract.
- **Capability boundary holds.** An MCP-submitted goal that ended up having Bootstrap try `web_fetch` directly (instead of delegating to fetcher) got `tool_result.success=false` with `capability denied: agent ... lacks NetworkAccess`. Exact capability the runtime promises to enforce. Nice to have this surface unprompted.

## What the run exposed

Seven bugs, ranging from packaging ergonomics to a load-bearing gap in the shipped TTL feature:

- **BUG #1 — `cargo deb` doesn't build the worker binary as a dependency.** The `.deb` assets in `crates/agentd/Cargo.toml` unconditionally list `target/release/aaos-agent-worker`. If the operator runs `cargo deb -p agentd` without a prior `cargo build -p aaos-backend-linux --bin aaos-agent-worker --release`, the pack fails with `Can't resolve asset`. Fix shape: either add a `pre-build` script to cargo-deb metadata that builds the worker, or make the worker asset conditional on a feature (e.g. `namespaced-agents`), or document the two-step in the packaging README. Real papercut for anyone trying to reproduce a release build.
- **BUG #2 — postinst doesn't `systemctl enable --now agentd`.** Fresh `apt install` leaves the unit in `disabled/inactive`. Debian convention is that services auto-start on install unless the admin intervenes. Fix shape: add `deb-systemd-helper enable agentd.service` + `systemctl start agentd.service || true` to `packaging/debian/postinst` under the `configure` case.
- **BUG #3 — `ReasoningScheduler` is invisible in logs.** `SchedulerView::complete` doesn't emit any tracing; the dispatcher loop logs only on dropped wakers. Operators observing LLM concurrency under load see only `aaos_llm::scheduled: inference permit acquired` lines (from the legacy `ScheduledLlmClient`). Can't confirm the new scheduler is actually in the critical path without a code read or a test-with-assertion. Fix shape: `tracing::debug!` on `acquire_slot` awarded and `SchedulerView::complete` entry/exit.
- **BUG #4 — Top-level LLM calls bypass the new scheduler.** The Planner's own `complete()` call (to produce the DAG) and Bootstrap's LLM calls go through `self.llm_client` directly, not through a `SchedulerView`. Only SUBTASK child agents get wrapped. So `AAOS_MAX_CONCURRENT_INFERENCE` is still entirely honored by `ScheduledLlmClient`'s semaphore — the new scheduler is an inner gate, not a replacement. The commit message for `d90189d` ("retired ScheduledLlmClient in a follow-up") is aspirational; the backstop is still the legacy path. Not a correctness bug — a honesty-in-docs bug. The roadmap should say "ReasoningScheduler wraps per-subtask LLM calls" not "replaces `ScheduledLlmClient`."
- **BUG #5 (load-bearing) — TTL hop decrement is never called.** `TaskTtl::decrement_hops` exists in `crates/aaos-core/src/task_ttl.rs` with unit tests, but **no call site anywhere in the codebase invokes it between subtasks**. `spawn_subtask` checks `ttl.max_hops == Some(0)` to refuse launch, but nothing decrements. Result: every subtask in a plan sees the same initial `max_hops` value. With `AAOS_DEFAULT_TASK_TTL_HOPS=1`, the 3-subtask chain (fetcher → analyzer → writer) runs to completion — each subtask reads `max_hops: Some(1)`, `1 != 0`, proceeds. The only way hops could fire is if the planner literally set `max_hops: 0` at plan creation, which it never does. **This ships the TTL-hops feature as non-functional** and neither the unit tests nor the inline `hop_exhaustion_fails_subtask_before_launch` test caught it — because that test literally hard-codes `max_hops: Some(0)`, bypassing the decrement path that doesn't exist. Needs a fix and a TEST that asserts hop count goes 3 → 2 → 1 → 0 → refused across a real 4-subtask chain.
- **BUG #6 — Wall-clock TTL doesn't apply to scaffolded roles.** `race_deadline` wraps only the LLM-powered-role branch of `spawn_subtask`; the scaffold-dispatch branch (for `fetcher` and future mechanical roles) returns earlier without any deadline wrapping. A long HTTP fetch (>60s) by the fetcher scaffold can't be killed. In practice `web_fetch` has its own 50 KB / short-timeout contract so this doesn't manifest today, but the design promise "TTL bounds runaway work" doesn't hold uniformly. Fix shape: wrap the scaffold runner call in `race_deadline` too, or declare scaffold roles TTL-exempt explicitly.
- **BUG #7 — `SubtaskTtlExpired` is not operator-visible.** `crates/agentd/src/cli/output.rs::is_operator_visible` whitelists `AgentSpawned`, `ToolInvoked`, `AgentExecutionCompleted`, `AgentLoopStopped`, `CapabilityDenied`, and failed `ToolResult`. `SubtaskTtlExpired` falls through to the default `_ => false`. Operators running with `AAOS_DEFAULT_TASK_TTL_WALL_CLOCK_S=3` see their analyzers disappear, replans happen silently, and Bootstrap terminally fails — with zero indication that TTL was responsible. `--verbose` exposes it (raw NDJSON), but default view hides the single most important event the new feature emits. Fix shape: add `SubtaskTtlExpired` to the whitelist and add a formatter branch that prints `ttl killed: {reason}` with a color hint.

## Lessons worth lifting

- **Inline unit tests can pass while the feature ships broken.** BUG #5 — hop decrement never called — had a unit test in `task_ttl.rs` covering the decrement function, a planner test covering `apply_ttl_fallback`, an inline integration test `hop_exhaustion_fails_subtask_before_launch` asserting the refuse-on-zero behavior. All green. None of them tested the full "plan arrives with Some(3), spawn_subtask A, decrement to Some(2), spawn_subtask B, decrement to Some(1), spawn_subtask C, refuse" chain. The test that would have caught it requires a realistic multi-subtask plan running through the real planner-plus-executor path — a test style we've resisted as "too close to integration." For anything with per-step mutation of a type (decrement, increment, checkpoint), the test MUST exercise the multi-step sequence, not just the step in isolation. If the test hard-codes the final state on entry (as `max_hops: Some(0)` does), it's a tautology test, not a behavioral test. Lift to `patterns.md` if a second case of this appears.
- **Operator-visible events are part of the feature, not decoration.** BUG #7 exists because the work to add `SubtaskTtlExpired` to `is_operator_visible` wasn't part of the plan's task list. Feature authorship has to include "how does the operator see this happen" as a first-class step, not a polish afterthought. Adjacent: Phase F-b sub-project 2 (dynamic model routing) will introduce `ModelRouted` or similar — the plan for that must include the operator-visibility task.
- **First production exercise finds what unit + CI never will.** We had 6 bugs that survived two-stage review + full green CI. All 6 surfaced in 20 minutes of driving the real binary on a real host. The subagent-driven loop caught correctness bugs between tasks, but didn't and can't catch "this whole feature is non-functional when threaded through the real code path." First-on-production e2e run should be part of every Phase F sub-project's Definition of Done, not an afterthought.
