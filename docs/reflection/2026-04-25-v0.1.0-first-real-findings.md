### Bug 13 (new) — agent stop races with in-flight tool invocation

The LLM emitted `file_write` at iter 18 (audit log shows the tool call with the full report content as the `content` parameter).  Within 2 ms of the audit log, the agent transitioned `running → stopping → stopped` and the subtask completed.  The actual `tool.invoke()` call appears to have been preempted: no file landed on disk, and the executor's "ToolUse arm complete — looping" log never fired for iter 18 (it had fired for iters 1-17).

Hypothesis: the workspace cleanup or a deadline check is signalling stop on the agent's command channel between (a) the audit log of the LLM response and (b) the for-loop over `tool_uses`.  Suspect: `scopeguard::guard` in `run_subtask_inline` (server.rs:590) firing because some upstream returned `Ok(...)` early.  Needs a follow-up debug pass with verbose tracing on the for-loop entry/exit.

Severity: High.  Means agents can be stopped between deciding to call a tool and actually calling it.  Visible failure mode is a missing output file; invisible failure mode is a tool that runs side-effects (`git_commit`, `file_write` to a sensitive path) and the call IS executed but the agent is stopped before recording it — both extremely bad outcomes.

# Bug Report: aaOS Production Crate Findings

## Finding 1: `max_invocations` constraint is silently non-functional for all tool calls

**File:** `/src/aaOS/crates/aaos-tools/src/invocation.rs` lines 237–275  
**Severity:** High — violates the documented security invariant (capability invocation limits)

### Description

`CapabilityToken` has a `max_invocations` constraint field and a `record_use()` method that increments `invocation_count`. The `permits()` method (line 178-189 of `capability.rs`) checks `invocation_count >= max_invocations` and returns `false` when exhausted. However, **`record_use()` is never called in the tool-invocation hot path**.

In `ToolInvocation::invoke()`:

1. **Lines 126–128 (capability check):** The code calls `self.capability_registry.permits(*h, agent_id, &required)` — which checks the threshold but does NOT increment `invocation_count`.

2. **Daemon path (lines 267–275):** The tool is looked up and invoked with `tool.invoke(input, &ctx).await`. Each individual tool (e.g., `file_write.rs` lines 60-63) calls `ctx.capability_registry.permits(...)` again — again without `record_use()`. After the tool returns, there is no call to `authorize_and_record()`.

3. **Worker path (lines 233–265):** Tokens are resolved via `resolve_tokens()` (which clones them with their current `invocation_count`) and forwarded to the worker. The worker inserts these into a **fresh, per-call** `CapabilityRegistry` (worker.rs line 245-249). The worker-side tool calls `permits()` on its own local registry (same problem), and even if it called `authorize_and_record()` the incremented count would be lost when the per-call registry is dropped.

**Result:** An agent with `max_invocations: Some(1)` for `ToolInvoke` (or any specific capability) can invoke that tool an unlimited number of times. The `invocation_count` field on `CapabilityToken` is completely dead code for enforcement purposes — it only ever increases in tests.

### Reproduction sketch

```rust
// In any integration test:
let registry = Arc::new(CapabilityRegistry::new());
let agent = AgentId::new();
let token = CapabilityToken::issue(
    agent,
    Capability::ToolInvoke { tool_name: "file_write".into() },
    Constraints { max_invocations: Some(1), rate_limit: None },
);
let handle = registry.insert(agent, token);
let invocation = build_invocation(registry, /* ... */);

// Call the tool 100 times — all succeed despite max_invocations=1
for _ in 0..100 {
    let result = invocation.invoke(
        agent, "file_write",
        json!({"path": "/tmp/x", "content": "x"}),
        &[handle],
    ).await;
    assert!(result.is_ok()); // passes — bug
}
// invocation_count should be 100 but max_invocations=1 was never enforced
```

### Proposed fix

In `ToolInvocation::invoke()`, after the tool-specific capability check passes (line 128) and **before** the tool is dispatched, call `authorize_and_record()` on the tool-invoke capability handle. This atomically checks AND records the use:

```rust
// In invocation.rs, after line 128 (has_permission check passes):
// Record the tool-invoke capability use atomically.
let _ = token_handles.iter().find(|h| {
    self.capability_registry
        .authorize_and_record(**h, agent_id, &required)
        .is_ok()
});
```

Additionally, each individual tool (file_read, file_write, etc.) should call `authorize_and_record()` instead of `permits()` when checking its path-specific capability, or daemon-side invocation should record use after the tool returns successfully.

For the worker path, invocation counts must be reported back to the daemon's registry, or the worker must receive a reference-counting scheme rather than a fresh per-call copy.

## Finding 2: Worker-side per-call `CapabilityRegistry` defeats capability revocation and exhaustion

**File:** `/src/aaOS/crates/aaos-backend-linux/src/worker.rs` lines 245–249  
**Severity:** High — allows use of revoked/exhausted tokens in some scenarios, and creates a confusing security model

### Description

When a tool call is routed to a confined worker, the daemon serializes the agent's current `CapabilityToken` structs (via `resolve_tokens()` in `capability_registry.rs` lines 203-218) and forwards them across the broker. The worker then rebuilds a **completely new, independent** `CapabilityRegistry` for each invocation:

```rust
// worker.rs lines 245-249
let per_call_registry = aaos_core::CapabilityRegistry::new();
let token_handles: Vec<aaos_core::CapabilityHandle> = capability_tokens
    .into_iter()
    .map(|t| per_call_registry.insert(agent_id, t))
    .collect();
```

This creates several problems:

1. **Revocation is invisible to the worker.** If the daemon revokes a token between the time `resolve_tokens()` serialized it and the worker is still processing, the worker's local copy continues to permit the operation. The worker's `CapabilityRegistry` is fresh per-call, so there is no mechanism for the daemon to signal a mid-invocation revocation.

2. **Token cloning defeats `invocation_count` tracking across worker calls.** Each worker call gets a fresh clone of the daemon's current token. If the daemon's token has `invocation_count: 3` and `max_invocations: 5`, and the worker uses it twice, both worker calls succeed — the count never propagates back to the daemon. The daemon's count stays at 3, the next worker call again gets `invocation_count: 3`.

3. **The per-call registry is never checked for agent identity on tokens that don't match.** `resolve_tokens()` already filters by `agent_id` (line 211-213), so mismatched tokens are dropped. But the worker re-inserts them with `per_call_registry.insert(agent_id, t)` — associating them with the worker's agent_id, which may differ from the token's original `agent_id` field. The `insert()` method does not verify consistency (it just stores what it's given).

### Reproduction sketch

```rust
// Daemon side:
let agent = AgentId::new();
let token = CapabilityToken::issue(agent,
    Capability::FileWrite { path_glob: "/data/*".into() },
    Constraints { max_invocations: Some(1), rate_limit: None },
);
let handle = registry.insert(agent, token);

// First worker call: resolve_tokens() clones the token (invocation_count=0)
// Worker gets a fresh copy and uses it. Daemon count stays 0.

// Now revoke on daemon:
registry.revoke(token.id);

// Second worker call: resolve_tokens() clones AGAIN (still invocation_count=0,
// and revoked_at was set on the daemon's copy — but the clone was taken
// BEFORE revoke in the serialization path... actually resolve_tokens clones
// the current state, so revoked_at IS propagated. But the check is only on
// the daemon's copy — the worker's local registry doesn't know about the
// revocation until the NEXT resolve_tokens call.)
//
// If the revoke happens AFTER resolve_tokens but WHILE the worker is
// still processing, the worker's copy remains unrevoked.
```

### Proposed fix

The token forwarding architecture needs a **session-level** capability view rather than per-call snapshots:

**Option A (simplest):** Send an updated `CapabilityRegistry` delta with each `InvokeTool` message that includes revocation state. The worker maintains a persistent registry per session (not per-call) and applies the delta before each new invocation.

**Option B (more thorough):** Instead of forwarding token structs, keep all capability checking on the daemon side. The worker executes the tool call with the daemon as a "capability proxy" — the worker sends the capability request to the daemon which checks and records use in one atomic operation, then allows or denies. This eliminates the distributed-token problem entirely.

**Minimum fix:** At minimum, the worker should call `authorize_and_record()` on its local registry and the daemon should decrement/refresh counts. But this still doesn't solve the revocation-during-invocation race. Option B is architecturally cleaner.

## Finding 3: `normalize_path` can silently produce an empty path component after root, allowing ambiguous absolute-path resolution

**File:** `/src/aaOS/crates/aaos-core/src/capability.rs` lines 475–491  
**Severity:** Medium — can lead to capability bypass under specific container configurations

### Description

The `normalize_path` function handles path normalization entirely lexically. There is a subtle edge case in how it handles paths with double slashes (`//`) and how it reconstructs absolute paths:

```rust
fn normalize_path(path: &str) -> String {
    let mut parts: Vec<&str> = Vec::new();
    for component in path.split('/') {
        match component {
            "" | "." => {}      // empty strings from split are skipped
            ".." => { parts.pop(); }
            other => parts.push(other),
        }
    }
    if path.starts_with('/') {
        format!("/{}", parts.join("/"))
    } else {
        parts.join("/")
    }
}
```

The issue: `split('/')` on `//` produces `["", "", ""]`. All three are empty and skipped, so `parts` stays empty. For an absolute path `path.starts_with('/')` → true, so the result is `format!("/{}", "")` → `"/"`. That's fine.

But consider `path = "/foo//bar"`. Split gives `["", "foo", "", "bar"]`. The empty string from `//` is skipped. Result: `"/foo/bar"`. Fine.

The real problem is when `normalize_path` is called within `canonical_for_match` (line 435) as the **first normalization step** before filesystem canonicalization. Consider a path like `/proc/self/root/../etc/passwd` — this is a symlink chain. The lexical normalization step produces `/etc/passwd` (correct), then `canonicalize()` resolves additional symlinks. So that's fine.

However, the more subtle issue is with the `glob_matches` function (line 403-415):

```rust
fn glob_matches(pattern: &str, path: &str) -> bool {
    if pattern == "*" { return true; }
    let canonical = canonical_for_match(path);
    if let Some(prefix) = pattern.strip_suffix('*') {
        let norm_prefix = normalize_path(prefix);
        canonical.starts_with(&norm_prefix)
    } else {
        let norm_pattern = normalize_path(pattern);
        norm_pattern == canonical
    }
}
```

**The bug:** When `pattern` is something like `/data/*`, `strip_suffix('*')` gives `Some("/data/")`. Then `normalize_path("/data/")`:
- split('/') → `["", "data", ""]`
- "" → skip
- "data" → push → parts=["data"]
- "" → skip
- starts_with('/') → true
- Result: `format!("/{}", "data")` → `"/data"`

Wait — the trailing `/` is **lost** because the empty string from the trailing slash is skipped. So `norm_prefix = "/data"`.

Then the check becomes: `canonical.starts_with("/data")`. This is fine for most cases, but consider:

- Pattern: `/data/*`
- Requested path: `/data-foo/bar.txt`

`canonical_for_match("/data-foo/bar.txt")` → let's say it exists and canonicalizes to `/data-foo/bar.txt`. Then `"/data-foo/bar.txt".starts_with("/data")` → **TRUE!** Even though `/data-foo` is not under `/data/`.

This is a **capability bypass**: a grant of `FileRead { path_glob: "/data/*" }` would incorrectly permit reading `/data-foo/bar.txt`, `/database/secret`, or any path starting with `/data` but not actually under `/data/`.

### Reproduction sketch

```rust
// Grant: /data/* should only allow /data/<anything>
let token = CapabilityToken::issue(
    agent,
    Capability::FileRead { path_glob: "/data/*".into() },
    Constraints::default(),
);

// These should ALL be denied but some are permitted:
assert!(!token.permits(&Capability::FileRead {
    path_glob: "/data-foo/secret.txt".into()
})); // FAILS! starts_with("/data") matches "/data-foo/..."

assert!(!token.permits(&Capability::FileRead {
    path_glob: "/dataXYZ/evil.sh".into()
})); // FAILS! same reason

assert!(!token.permits(&Capability::FileRead {
    path_glob: "/database/dump.sql".into()
})); // FAILS! "/database/dump.sql".starts_with("/data") is FALSE
     // because "/database" doesn't start with "/data" — so this one
     // actually works correctly. Only sibling prefixes match.
```

Wait, let me re-check. `"/data-foo/secret.txt"` — does it start with `"/data"`? Yes: `"/data-foo/secret.txt"[..5]` = `"/data"`. So the glob prefix check is `"/data-foo/secret.txt".starts_with("/data")` → TRUE. This is incorrect — `/data-foo` is not under `/data/`.

The correct comparison should be `canonical.starts_with(&norm_prefix) && canonical.as_bytes().get(norm_prefix.len()) == Some(&b'/')` — ensuring the prefix boundary falls on a path separator.

### Proposed fix

In `glob_matches`, when matching a prefix glob (pattern ending with `*`), ensure the canonical path either equals the prefix exactly or has a separator after it:

```rust
fn glob_matches(pattern: &str, path: &str) -> bool {
    if pattern == "*" { return true; }
    let canonical = canonical_for_match(path);
    if let Some(prefix) = pattern.strip_suffix('*') {
        let norm_prefix = normalize_path(prefix);
        // Ensure the prefix is followed by '/' or is the full path
        // to prevent /data/* matching /data-foo
        canonical.starts_with(&norm_prefix)
            && (canonical.len() == norm_prefix.len()
                || canonical.as_bytes().get(norm_prefix.len()) == Some(&b'/'))
    } else {
        let norm_pattern = normalize_path(pattern);
        norm_pattern == canonical
    }
}
```

This fix ensures that a grant of `/data/*` matches `/data/file.txt` and `/data/subdir/file.txt` but not `/data-foo/secret.txt` or `/database/evil.sh`.
