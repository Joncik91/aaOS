## Bug 13 — diagnosis path

Sonnet sub-agent traced the tokio cancellation chain from `exec_task.abort()` (server.rs streaming handler) through `race_deadline`'s `tokio::select!` down to the `invoke_tool(...).await` in the executor's ToolUse arm.  Found:

1. The verbose audit log (`--- TOOL CALL: file_write ---`) fires before `.await`, so it's always recorded.
2. The actual `tool.invoke()` is a `.await` — when the parent task is aborted, the future is dropped at this exact suspension point.
3. The `_guard` (`scopeguard`) in `run_subtask_inline` fires on drop, calling `stop_sync(agent_id)` → `running → stopping → stopped` state transition.
4. The 1.9 ms gap between the audit log and the state transition is the time for tokio to deliver the cancellation and run the synchronous scopeguard drop.

Two possible triggers: (a) client-disconnect at the streaming handler (write_ndjson fails or RecvError::Closed), or (b) wall-clock TTL fired in `race_deadline`.  Both cancel mid-tool-call.  The fix is the same regardless of trigger: a 500 ms drain window before the abort lets in-flight tool invocations complete.

Fix at `34b018e`:
- 500 ms drain via `tokio::time::timeout(&mut exec_task)` at all 4 `exec_task.abort()` sites (plan branch + direct branch, write-failure + RecvError::Closed cases).
- `tracing::warn!` in `race_deadline` when the TTL branch fires, so post-mortem can distinguish TTL-triggered drops from client-disconnect-triggered drops.

## Bug 17 — workspace path mismatch (new finding from same run)

The 10.9 KB report DID land on disk — but at `/var/lib/aaos/workspace/<run-id>/output.md`, not at `/data/findings.md` where the goal text said.  Operator with shell access on the droplet found the file by `find` only because they knew where to look.

Root cause: `crates/agentd/src/server.rs::inline_direct_plan` hardcoded `workspace: "{run}/output.md"` in the subtask params.  The generalist's `packaging/roles/generalist.yaml` system prompt has this clause: *"If your message includes a 'workspace' path, that is where your primary output must go.  Call file_write(path=<workspace>, ...)."*  The workspace param is treated as authoritative; the goal text's "write to /data/findings.md" is overridden.

The same misdelivery happened at the Bug 13 verification step too — the generalist tried to `file_read` `/var/lib/aaos/workspace/.../output.md` (the path it had just written to) and got `capability denied: lacks FileRead { path_glob: "/var/lib/aaos/workspace/..." }` because the generalist's read caps are `/data/*` and `/src/*`, not workspace-internal paths.  The agent recognised the absurdity of "I just wrote it but can't read it" in the audit stream — but by then the operator-promised file path was empty.

Severity: medium-high.  Same silent-misdelivery class as Bug 9 was, at the workspace-path layer.  Direct-mode runs always lose their output to a workspace-internal path the operator can't easily find or read.

Fix in `77bbe9d`: omit the workspace param entirely from `inline_direct_plan`.  The generalist's "if no workspace, follow the task description" fallback path triggers, and the LLM writes to whatever path the goal text named.  Risk: LLM picks a path the generalist's caps don't cover — produces a clean capability-denied error rather than silent misdelivery, which is the better failure mode.  Also tightened the EXECUTION CONTRACT block in the wrapped goal to explicitly say "the operator-specified path" instead of the old "or {workspace} if none" hedge.

## What the agent's report contained — 3 candidate bug findings

The 10.9 KB report has three new findings the agent thinks are real bugs.  Need triage before deciding whether to ship fixes:

1. **TOCTOU race in `CapabilityRegistry::narrow`** — `crates/aaos-core/src/capability_registry.rs:61`.  Claim: race between checking the parent token's revocation state and creating the narrowed child handle.
2. **Seccomp allowlist permits `clone3` without argument filtering** — `crates/aaos-backend-linux/src/seccomp_compile.rs:182`.  Claim: enables worker-to-host process injection.  The source comment at line 19 self-acknowledges "simplify: clone3 is allowed unconditionally," so the smell is real and known; the question is severity.
3. **`BudgetTracker::maybe_reset` TOCTOU** — `crates/aaos-core/src/budget.rs:105`.  Claim: race between the time-elapsed check and the reset, allowing a small window where an agent could exceed its budget.

All three reference real code at the cited file:line.  Whether they're load-bearing security defects vs. theoretical races needs careful read-through against the existing locking structure.  Queued as Bugs 18/19/20 in CHANGELOG `[Unreleased]` for triage.  Not shipped in v0.1.2.
