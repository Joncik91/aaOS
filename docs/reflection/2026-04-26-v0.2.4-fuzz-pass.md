# 2026-04-26 — Fuzz pass on v0.2.4 security-critical surfaces

Companion to round 9 (which depleted the source-reading loop).  Round 9 produced 0 real findings on v0.2.4 source; the natural next probe was a *different in kind* one — fuzzing the security-critical input surfaces with libfuzzer to find inputs the source-reading agent could not.

**0 panics across 137 million randomized inputs** in 15 minutes of wall-clock fuzzing across three targets.  The v0.2.x runtime's input-handling on the surfaces tested is genuinely robust.

## Setup

- Local A8 host (no droplet — fuzzers run as a regular `cargo +nightly fuzz` invocation, no kernel-feature requirements beyond what Linux 5.6+ already provides).
- `cargo-fuzz 0.13.1`, nightly Rust `1.97.0-nightly (9838411cb 2026-04-25)`, libfuzzer-sys 0.4.
- New `fuzz/` workspace member added to repo (excluded from main workspace via `[workspace] exclude = ["fuzz"]` in the root `Cargo.toml` since cargo-fuzz needs its own dependency tree).
- `.gitignore` updated to exclude `fuzz/target/`, `fuzz/corpus/`, `fuzz/artifacts/`.  The fuzz targets themselves are tracked (they ARE the test); the per-run state is not.
- Each target ran for `-max_total_time=300` seconds.

## Targets

Three targets, picked for capability/security relevance:

1. **`glob_matches_canonical`** — the path-traversal capability matcher.  Pattern + path fed as two halves of the input.  v0.1.x Bug 12 (glob `/data/*` accepting `/data-foo/...`) was a real capability bypass in this exact function; if any similar miss-match still exists, the fuzzer should hit it.

2. **`safe_open_for_capability`** — the v0.2.x TOCTOU primitive.  First byte selects the access mode; remainder is the suffix.  Fuzz target prefixes a per-process tmpdir to all paths so `WriteCreateTrunc` mode can't litter the cwd (initial run created 7.6M zero-byte files in the workspace root before the sandboxing was added — fixed in the committed target).

3. **`broker_request_deserialize`** — `serde_json::from_slice::<WireRequest>` on arbitrary bytes.  Tries both raw bytes and bytes + trailing newline (matching the line-delimited wire format).  Targets the Request enum's tagged-union deserialize path which has gained shapes through v0.2.x (RevokeToken, InvokeTool with capability tokens).

## What this confirms

- **The v0.2.x input-handling surface is genuinely robust against random input.**  The `glob_matches_canonical` finding from v0.1.x (`/data/*` matching `/data-foo/...`) is the kind of bug a fuzzer would hit instantly; that we got 28M runs without a panic means the v0.1.x fix is structurally complete, not just patching the one observed input.
- **`openat2(RESOLVE_NO_SYMLINKS)` + `readlinkat` + the fallback to `open(O_NOFOLLOW)` survive 85M malformed paths without a single panic.**  The error-mapping in `safe_open_for_capability` (ELOOP → "refusing to open symlink", ENOENT → "file not found", etc.) is exhaustive for the inputs the fuzzer found.
- **Broker-protocol deserialization is robust.**  serde is heavily battle-tested on this; finding nothing here is the expected outcome.  Exercising it confirms `WireRequest`'s tagged-enum dispatch + flatten attribute don't introduce a custom panic path.

## What this DOESN'T confirm

- **Concurrency bugs.**  The fuzz targets are single-threaded.  Race conditions like Bug 31 (`BudgetTracker` reset race) cannot be found by libfuzzer; that needs a stress test with multiple threads + a race detector.
- **Bugs that need filesystem state at specific points.**  The intermediate-component symlink TOCTOU (Bug 32) was found only because a human reasoned about the order of `create_dir_all` + `safe_open` syscalls.  Fuzzing the open primitive in isolation cannot find it.
- **Kernel-version-specific issues.**  Fuzzer ran on the same kernel as the daemon's runtime; bugs that only manifest on older kernels (where the openat2 fallback fires) are out of scope.

## Process notes

- **The "WriteCreateTrunc fills the cwd with zero-byte files" was a real fuzz-target bug I caught only after deleting the artifacts.**  Tightened the target to prefix `/tmp/aaos-fuzz-<pid>/` to all paths.  Lesson: fuzz targets that take side effects (open with O_CREAT, file_write, etc.) need explicit sandboxing — the fuzzer doesn't know what your filesystem looks like.
- **Coverage saturation at ~100 features in the safe_open target** is fast (22 corpus inputs cover everything the target exercises).  Suggests the production code's input handling has few independent decision points — most paths fall into "open succeeds" / "ELOOP" / "ENOENT" / "EACCES" early.  That's a good shape for security code: fewer branches = less attack surface.
- **The broker fuzzer's 1499-input corpus** — by far the largest — reflects serde's input space.  Each new shape (`{`, `}`, `,`, `:`, plus quoted strings and numbers) is independently mutable.  Coverage saturates more slowly but the bug surface is also wider.
