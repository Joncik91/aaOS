## What the first production exercise exposed

Two bugs and one design-layer gap. This is exactly what `patterns.md`'s "First on-production exercise finds what unit + CI never will" predicts — the lesson applies to bug-fix commits too, not just feature ships.

### BUG #1 — every worker-routed tool call failed with `no broker session for agent <id>`

**Symptom.** On the first submit, every `file_write` with the intended `[worker]` routing failed immediately:
```
[13:20:02] 6cbb7223    tool: file_write [worker] {...}
[13:20:02] 6cbb7223    tool FAILED: file_write — ipc error: no broker session for agent 6cbb7223-...
```

**Root cause.** `execute_agent_for_subtask` in `crates/agentd/src/server.rs` spawns inline subtask agents — it calls `AgentExecutor::run` directly instead of going through `AgentBackend::launch`. So `NamespacedBackend::session(agent_id)` returned `None` for every plan-executor subtask, because no worker was ever launched for that agent id.

**Fix shipped** (commit `7a017f6`):
- `WorkerHandle::invoke_over_worker` now returns `Result<Value, WorkerInvokeError>` where the error distinguishes `NoSession` (agent has no worker — legitimate for inline subtasks) from `Transport(String)` (real broker failure).
- `ToolInvocation::invoke` catches `NoSession`, falls back to daemon-side execution, and records `ToolExecutionSurface::Daemon` on the audit event so operators see the honest truth.
- Moved the `ToolInvoked` audit emit to AFTER the call so `execution_surface` reflects what actually ran, not the intent.
- Unit test `worker_no_session_falls_back_to_daemon` pins the behaviour: `NoSession` handle → daemon-side call succeeds → audit records `Daemon`.

**What it teaches.** Two execution paths (inline vs spawn_agent) existed before sub-project 3; only spawn_agent goes through `backend.launch` and gets a worker session. The sub-project 3 design wrote this down ("the ToolInvocation fork routes calls to worker when backend_kind == namespaced") without noticing that half of all agents don't have workers. Five unit tests and three integration tests passed without catching it because they only exercised the path where a worker session exists.

### BUG #2 — worker-side tool execution re-checks capabilities inside the worker and fails

**Symptom.** After BUG #1's fix, the `invoke_tool_file_read_roundtrips` integration test still failed on the droplet:
```
file_read over broker should succeed inside Landlock scope:
Runtime("capability denied: agent <id> lacks FileRead { path_glob: \"/lib/x86_64-linux-gnu\" }
       : file_read not permitted for path: /lib/x86_64-linux-gnu")
```

**Root cause.** `FileReadTool::invoke` does its own `ctx.capability_registry.permits(handle, agent_id, &required)` check (crates/aaos-tools/src/file_read.rs:67). The worker's `InvocationContext` has an **empty** `CapabilityRegistry` and `tokens: vec![]` — per T6 design, which said "daemon-side `ToolInvocation` has already done the capability check; worker context exists only to satisfy the trait signature." But the tools themselves encode their own re-check, so the worker's empty registry + empty tokens denies every call.

**Not fixed in v1.** Fixing this cleanly is a design layer — either:
- (a) Factor out a `Tool::invoke_no_cap_check()` variant and have the worker call the no-check flavor.
- (b) Pass `(CapabilityHandle, CapabilityToken)` pairs across the broker and let the worker rebuild a minimal registry just for the invoking agent's grants.
- (c) Have the daemon pre-bind the capability check to the tool call (curry it out) so the worker never sees a capability registry at all.

Each of those is a non-trivial refactor. v1 of sub-project 3 ships the confinement substrate (broker stream routing, worker registry, surface tag, Landlock as second gate) — but until this layer is fixed, the *only* path where worker-side execution actually works end-to-end is the `file_read` → Landlock-denial negative test (T10), which passes because the tool's capability check permits `/etc/shadow` (the test grants wildcard) and Landlock provides the real denial.

**Evidence the substrate works.** `cargo test -p aaos-backend-linux --lib invoke_tool -- --ignored --nocapture` on the droplet:

```
running 2 tests
... invoke_tool_landlock_denial ... ok
... invoke_tool_file_read_roundtrips ... FAILED
```

T10 (Landlock denial) passed: the call crossed the broker, entered the worker, the worker attempted the tool, the *tool's* capability check ALSO passed (wildcard grant), and then **Landlock denied the filesystem access** — the error surfaced back as a `ToolDenied`-shaped error string. Exactly the defense-in-depth the design promised.

T9 (file_read positive case) failed on the capability re-check — a ways before Landlock got a vote.

### Design-layer gap — plan-executor subtasks never get worker isolation

Following from BUG #1's root cause: today's plan-executor path spawns all subtasks inline via `execute_agent_for_subtask`. That's faster than spawning a namespaced child per subtask, but it means **every plan-executor agent runs daemon-side** — sub-project 3's confinement activates only for explicit `spawn_agent` tool calls.

Two architectural alternatives:
- Spawn every plan-executor subtask through `backend.launch` (adds ~100ms per subtask for namespace + Landlock setup, multiplied by N subtasks per plan).
- Keep inline spawning but document it as "sub-project 3 confines only agents launched via spawn_agent." The canonical-goal DAG uses the plan executor, so in practice confinement activates only when nested agents spawn sub-agents via the tool.

v1 of sub-project 3 takes the second path by default (the `NoSession` fallback ensures inline subtasks still work) but the scope needs to be named honestly in architecture.md — sub-project 3 **does not** confine plan-executor subtasks today.

## Lessons worth lifting (or re-confirming)

**"First on-production exercise finds what unit + CI never will" — re-confirmed.** Sub-project 2's e2e passed first-try because its internal shape was unit-testable end-to-end (no inline vs. spawn_agent path bifurcation). Sub-project 3 has a genuinely split world — inline subtasks live daemon-side forever, only `spawn_agent`-launched children get worker sessions — and CI + unit tests couldn't exercise that split because both tests used the spawn_agent path. The production run hit the inline path first because the plan executor IS the inline path.

**New pattern candidate — "test the path the canonical goal actually uses."** T9 and T10 (integration tests) exercised the worker path correctly, but the canonical-goal benchmark goes through the plan executor, which doesn't use that path at all. The DoD should have explicitly required: "run the canonical goal and observe at least one tool call carrying `[worker]`." That's what T14 asked for; T9 + T10 were not sufficient substitutes. Lifted to patterns.md.

**Design doc honesty wins again.** The design said "tool invocations route through the broker" without checking which agents actually have brokers. Sub-project 2's scope notes warned about this pattern ("Gap 1's original description oversold what `SchedulerView` wrapped"). Sub-project 3 fell into the same trap from a different angle — the scope on "which agents are confined" needs to land in architecture.md before the ink dries on the commit.
